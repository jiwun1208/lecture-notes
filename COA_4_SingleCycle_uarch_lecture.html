Basic Microarchitectures of MIPS ISA: Single-Cycle Design

이 강의는 MIPS ISA(명령어 집합 구조)를 하드웨어로 실제 구현하는 방법 중 가장 기본적인 싱글사이클 마이크로아키텍처(Single-Cycle Microarchitecture)를 다룬다.
강의 개요

컴퓨터 구조(Computer Organization and Architecture) 과목에서 ISA를 배운 후, 그 ISA를 실제 디지털 회로로 구현하는 과정을 학습한다. 이번 장에서는 가장 단순한 구현 방식인 싱글사이클 설계를 처음부터 단계별로 구축해 나간다.

싱글사이클 설계란, 하나의 명령어를 단 한 클럭 사이클(clock cycle) 안에 완전히 실행하는 방식이다. 구현이 단순하지만 성능 면에서 한계가 있으며, 이후 강의에서 배울 파이프라인(Pipeline) 설계의 기초가 된다.

복습: RISC vs. CISC

MIPS는 RISC 방식의 ISA이다. 구현에 앞서 RISC와 CISC의 장단점을 복습하고, ISA 수준에서 성능과 전력을 논하기 어려운 이유를 이해한다.
상세 설명

RISC(Reduced Instruction Set Computer)는 명령어의 종류를 단순하게 유지하는 방식이다.

  • 장점: 디코더(Decoder) 설계가 단순하다. 하드웨어 세부 사항이 명령어에 드러나므로 컴파일러가 최적화하기 쉽다.
  • 단점: 하나의 작업을 수행하는 데 더 많은 명령어가 필요해 코드 크기가 커지는(Code Bloating) 문제가 있다.

CISC(Complex Instruction Set Computer)는 하나의 명령어가 복잡한 연산을 수행하는 방식이다.

  • 장점: 복합 명령어 덕분에 코드 크기가 작다(Compact code size).
  • 단점: 명령어 내부에 세부 동작이 숨겨져 있어 컴파일러 최적화가 어렵다. 디코더 설계가 복잡하며, 이를 마이크로코드(Microcode)로 부분적으로 해결한다.
핵심 질문: ISA 수준에서 RISC와 CISC의 전력(Power)과 성능(Performance)을 직접 비교할 수 있는가? → 아니다. 성능과 전력은 ISA 자체가 아니라 마이크로아키텍처(구현 방식)에 크게 좌우된다. 이것이 바로 마이크로아키텍처를 공부하는 이유이다.
핵심 용어
영어한국어설명
RISC 축소 명령어 집합 컴퓨터 단순한 명령어들로 구성. MIPS, ARM, RISC-V 등
CISC 복합 명령어 집합 컴퓨터 복잡한 명령어 지원. x86 등
Microcode 마이크로코드 복잡한 CISC 명령어를 더 단순한 내부 연산으로 분해하는 방법
Code Bloating 코드 비대화 RISC에서 같은 작업에 더 많은 명령어가 필요한 문제

CPU는 근본적으로 유한 상태 기계(FSM)이다

ISA는 추상적인 유한 상태 기계(FSM)를 정의한다. 입력은 명령어, 상태는 아키텍처 상태, 다음 상태 로직은 명령어 실행이다.
상세 설명

유한 상태 기계(Finite State Machine, FSM)란 유한한 수의 상태(State)를 가지며, 입력에 따라 상태가 전이(Transition)되는 수학적 모델이다. CPU를 FSM으로 보면:

  • 입력(Input): 현재 실행할 명령어(Instruction)
  • 상태(State): 아키텍처 상태(Architectural States) — 프로그램 카운터(PC), 범용 레지스터(GPR), 메모리 내용 등
  • 다음 상태 로직(Next State Logic): 명령어의 실행(Execution) — 현재 명령어를 실행하면 아키텍처 상태가 갱신됨

비유하자면, CPU는 "명령어 카드를 한 장씩 받아 읽고, 그에 따라 칠판(레지스터, 메모리)의 내용을 바꾸는 기계"와 같다.

핵심: ISA는 추상적(abstract)이다. 같은 FSM을 구현하더라도 전력 효율적(power-efficient)이거나 면적 효율적(area-efficient)인 등 다양한 구현이 가능하다. 이것이 마이크로아키텍처가 다양한 이유이다.
핵심 용어
영어한국어설명
FSM 유한 상태 기계 유한한 상태들과 상태 전이 규칙으로 정의되는 계산 모델
Architectural State 아키텍처 상태 PC, 레지스터 파일, 메모리 등 ISA가 정의하는 프로그래머가 볼 수 있는 상태

왜 마이크로아키텍처가 필요한가?

아키텍처(ISA)는 시스템의 기능만 정의할 뿐이다. 이를 실제 하드웨어로 만들려면 디지털 회로 수준의 구현, 즉 마이크로아키텍처가 필요하다.
상세 설명

아키텍처(Architecture, ISA)는 "무엇을 하는가"를 정의한다. 예를 들어 "ADD 명령어는 두 레지스터의 값을 더해 결과를 세 번째 레지스터에 저장한다"는 것은 ISA의 정의이다.

마이크로아키텍처(Microarchitecture)는 "어떻게 하는가"를 정의한다. 즉, 그 ADD 연산을 실제로 수행하기 위한 회로(가산기, MUX, 레지스터 파일 등)의 구성과 연결 방식이다.

같은 ISA(예: x86)를 구현하더라도, AMD의 Opteron 마이크로아키텍처와 Intel의 Core i7(Nehalem) 마이크로아키텍처는 완전히 다른 내부 설계를 가진다. 성능, 전력 소모, 칩 면적 등이 마이크로아키텍처에 따라 달라진다.

예시: x86 ISA의 서로 다른 구현
구현제조사특징
AMD Opteron AMD AMD의 서버용 마이크로아키텍처
Intel Core i7 (Nehalem) Intel Intel의 고성능 마이크로아키텍처

두 프로세서 모두 같은 x86 프로그램을 실행할 수 있지만, 내부 구조는 완전히 다르다.

마이크로아키텍처 = ISA의 디지털 수준(Digital Level) 구현. ISA는 "계약서", 마이크로아키텍처는 "실제 건축물"이라고 생각하면 된다.

아키텍처 상태의 마이크로아키텍처 (1)

프로그램 카운터(PC)와 레지스터 파일(Register File)은 플립플롭(Flip-Flop) 기반 레지스터로 구현된다. 읽기는 즉시(Probing), 쓰기는 클럭의 양의 에지(Positive Edge)에서 수행된다.
상세 설명

ISA가 정의하는 아키텍처 상태를 실제 하드웨어로 구현하는 방법을 알아보자.

프로그램 카운터(PC)와 레지스터 파일(Register File)은 플립플롭(Flip-Flop)으로 만든 레지스터로 구현된다. 각 레지스터는 레지스터 인덱스(Register Index)로 주소를 지정한다.

  • 읽기(Read)는 프로빙(Probing) 방식으로 수행된다. 즉, 주소를 주면 즉시(combinational하게) 해당 값이 출력된다. 클럭 에지를 기다릴 필요가 없다.
  • 쓰기(Write)는 클럭의 양의 에지(Positive Edge)에서 수행된다. 즉, 클럭 신호가 0에서 1로 올라가는 순간에만 값이 갱신된다.

이 비대칭적 동작이 중요하다. 한 사이클 안에서 레지스터를 먼저 읽고, 연산을 수행한 뒤, 다음 클럭 에지에서 결과를 쓸 수 있기 때문이다.

핵심 용어
영어한국어설명
Flip-Flop 플립플롭 1비트를 저장하는 가장 기본적인 순차 논리 소자. 클럭 에지에서 값이 갱신됨
Register File 레지스터 파일 여러 개의 레지스터를 묶어 인덱스로 접근할 수 있게 한 구조
Positive Edge 양의 에지 클럭 신호가 0에서 1로 전환되는 순간
Read-at-once (Probing) 즉시 읽기 주소 입력 시 조합 논리적으로 즉시 데이터가 출력되는 방식
읽기는 조합 논리(Combinational) → 즉시 출력, 쓰기는 순차 논리(Sequential) → 클럭 에지에서만. 이 차이를 반드시 기억하자.

아키텍처 상태의 마이크로아키텍처 (2)

메모리는 SRAM, DRAM 등 다양한 옵션이 있다. 이 장에서는 분석 편의를 위해 레지스터와 같은 동작 방식의 메모리와 명령어/데이터 분리 메모리를 가정한다.
상세 설명

실제 메모리는 SRAM(Static RAM), DRAM(Dynamic RAM) 등 다양한 기술로 구현되며, 각각 속도와 비용이 다르다. 하지만 이 장에서는 분석을 단순화하기 위해 두 가지를 가정한다:

  • 가정 1 — 레지스터와 같은 메모리 동작: 읽기는 즉시(Read-at-once), 쓰기는 한 사이클 내에 완료(Write-in-a-cycle). 실제 DRAM은 수십~수백 사이클이 걸리지만, 여기서는 무시한다.
  • 가정 2 — 두 개의 분리된 메모리: 명령어 메모리(Instruction Memory)와 데이터 메모리(Data Memory)를 분리한다. 이를 통해 같은 사이클에 명령어를 읽으면서 데이터도 접근할 수 있다.
실제 상황 주의: 실제로 데이터 이동 명령어(Load/Store)는 메모리 접근으로 인한 높은 지연(Latency) 오버헤드를 수반한다. 이 장의 가정은 교육 목적의 단순화임을 기억하자.
핵심 용어
영어한국어설명
SRAM 정적 RAM 빠르지만 비싸고 면적이 큼. 캐시 메모리에 사용
DRAM 동적 RAM 느리지만 저렴하고 고밀도. 주 메모리에 사용

명령어 처리의 5대 단계

모든 명령어는 IF → ID → EX → MEM → WB의 5단계를 거쳐 처리된다. 이 단계 구분은 프로세서 설계의 기본 골격이다.
상세 설명

명령어가 CPU에서 처리되는 과정을 5단계로 나누면 다음과 같다:

  1. IF (Instruction Fetch): PC가 가리키는 주소에서 명령어를 가져온다. 명령어 메모리에 주소(PC)를 넣으면 해당 명령어가 출력된다.
  2. ID (Instruction Decode): 가져온 명령어를 해석(디코딩)한다. opcode, rs, rt, rd, immediate 등의 필드를 분리하고, 레지스터 파일에서 필요한 값을 읽는다.
  3. EX (Execute): ALU를 사용하여 연산을 수행한다. 덧셈, 뺄셈, 논리 연산, 주소 계산 등이 여기서 이루어진다.
  4. MEM (Memory Access): 필요한 경우 데이터 메모리에 접근한다. Load는 메모리에서 읽고, Store는 메모리에 쓴다. ALU 연산 명령어는 이 단계를 그냥 통과한다.
  5. WB (Write Back): 연산 결과 또는 메모리에서 읽은 값을 목적지 레지스터에 기록한다.
데이터패스 흐름도
PC → [Inst Memory] → Instruction ↓ ↓ address [Register File] → r1, r2 ↓ [ALU] → result/address ↓ [Data Memory] → data ↓ Write Back to Register File

위 다이어그램은 아직 추상적인 수준이다. 여러 세부 사항(MUX, 제어 신호 등)이 생략되어 있으며, 이후 페이지에서 하나씩 구체화해 나간다.

목차: 싱글사이클 마이크로아키텍처

이제 본격적으로 싱글사이클 마이크로아키텍처의 개념과 구현에 들어간다.
상세 설명

앞서 ISA와 마이크로아키텍처의 관계, 아키텍처 상태의 하드웨어 구현, 명령어 처리 5단계를 복습했다. 이제부터는 싱글사이클 마이크로아키텍처를 처음부터(from scratch) 구축하는 과정을 시작한다.

Page 11

처음부터 시작하는 구현(Start from Scratch)

가장 단순한 디지털 시스템은 조합 논리(Combinational Logic)와 플립플롭(Flip-Flop)의 조합이다. 이 구조를 마이크로아키텍처의 뼈대로 사용한다.
상세 설명

디지털 시스템의 가장 기본적인 구조를 생각해 보자:

  • 조합 논리(Combinational Logic): 입력이 주어지면 즉시 출력을 계산하는 회로. 설계 대상의 핵심 기능 대부분이 여기에 들어간다.
  • 플립플롭(Flip-Flop): 상태를 유지하는 소자. 클럭의 양의 에지에서 입력 값을 저장한다.

이 구조에서, 입력과 출력은 매 클럭의 양의 에지(Positive Edge)마다 주입(Inject)되고 배출(Eject)된다. 즉:

  1. 양의 에지 → 플립플롭에 새 값이 저장됨
  2. 플립플롭 출력이 조합 논리의 입력으로 들어감
  3. 조합 논리가 결과를 계산함 (시간이 소요되지만 클럭 에지 전에 완료되어야 함)
  4. 다음 양의 에지 → 결과가 다시 플립플롭에 저장됨
핵심 질문: 이 "조합 논리 + 플립플롭" 템플릿을 마이크로아키텍처의 프레임으로 사용할 수 있는가? → YES! 이것이 싱글사이클 마이크로아키텍처의 출발점이다.
조합 논리 안에 ALU, 디코더, MUX 등의 기능 블록이 들어가고, 플립플롭에는 PC와 레지스터 파일이 대응된다고 생각하면 된다.

싱글사이클 마이크로아키텍처

싱글사이클 마이크로아키텍처는 조합 논리 부분에 마이크로아키텍처 전체를 매핑하여, 하나의 명령어를 한 클럭 사이클에 실행하는 가장 단순한 구현 방식이다.
상세 설명

앞 페이지의 "조합 논리 + 플립플롭" 템플릿에 마이크로아키텍처를 매핑하면:

  • 조합 논리 부분에 명령어 메모리, 디코더, 레지스터 파일 읽기, ALU, 데이터 메모리 등 모든 기능이 들어간다.
  • 매 클럭 사이클의 양의 에지에서 새로운 명령어 실행이 시작된다.
  • 한 사이클 안에 IF → ID → EX → MEM → WB가 모두 완료된다.

예를 들어, 첫 번째 클럭 사이클에서 ADD 명령어가 실행되고, 두 번째에서 LDW(Load Word), 세 번째에서 JAL(Jump and Link)이 실행되는 식이다.

R-type, I-type, J-type 등 모든 종류의 명령어가 한 사이클 안에 실행되어야 하므로, 클럭 주기는 가장 느린 명령어에 맞춰야 한다.

싱글사이클 마이크로아키텍처 = 가장 단순한 구현 방식. 장점은 구현의 단순성이고, 단점은 모든 명령어가 가장 느린 명령어의 속도에 맞춰져 성능이 낭비된다는 것이다.
핵심 용어
영어한국어설명
Single-Cycle 싱글사이클 모든 명령어를 정확히 한 클럭 사이클에 실행하는 방식
Combinational Logic 조합 논리 상태를 저장하지 않고 입력에 대해 즉시 출력을 생성하는 회로
Clock Cycle 클럭 사이클 클럭 신호의 한 주기. 싱글사이클에서는 명령어 하나의 실행 시간
성능 한계: Load 명령어가 5개 단계를 모두 사용하므로 가장 느리다. 단순한 ADD도 Load와 같은 시간을 소비해야 한다. 이 비효율을 해결하는 것이 이후에 배울 멀티사이클 및 파이프라인 설계이다.

구현 전 복습: MIPS 명령어 형식

MIPS에는 R-type, I-type, J-type 세 가지 명령어 형식이 있다. 모든 명령어는 32비트이며, 워드(Word) 단위로 정렬되어야 한다.
R-Type 명령어 형식

3개의 레지스터 오퍼랜드를 사용하는 형식 (ADD, SUB, AND, OR, SLT 등):

| 0 (fixed) | rs | rt | rd | shamt | funct | | 6-bit | 5-bit | 5-bit | 5-bit | 5-bit | 6-bit |
  • opcode: 항상 0 (R-type 식별)
  • rs, rt: 소스 레지스터
  • rd: 목적지 레지스터
  • shamt: 시프트 양(Shift Amount)
  • funct: 실제 연산 종류를 결정 (ADD=0x20, SUB=0x22 등)
I-Type 명령어 형식

2개의 레지스터와 1개의 즉시값(Immediate) 오퍼랜드를 사용하는 형식 (ADDI, LW, SW, BEQ 등):

| opcode | rs | rt | immediate (16-bit) | | 6-bit | 5-bit | 5-bit | 16-bit |
  • opcode: 명령어 종류를 결정
  • rs: 소스 레지스터
  • rt: 목적지 레지스터(ALU) 또는 소스(Store)
  • immediate: 16비트 상수값. 32비트로 부호 확장하여 사용
J-Type 명령어 형식

1개의 즉시값 오퍼랜드를 사용하는 형식. 주어진 주소로 점프 (J, JAL):

| opcode | immediate (26-bit) | | 6-bit | 26-bit |
  • opcode: J 또는 JAL을 구분
  • immediate: 26비트 점프 대상 주소
워드 정렬(Word Alignment): 모든 명령어는 32비트(4바이트) 단위로 정렬되어야 한다. 따라서 명령어 주소의 하위 2비트는 항상 00이다. 이는 점프/분기 명령어에서 주소 계산 시 2비트 오프셋 주소 지정(2-bit offset addressing)으로 활용된다.

목차: 비제어 흐름 명령어 구현

이제 비제어 흐름 명령어(Non-Control Flow Instructions), 즉 ALU 연산과 Load/Store 명령어를 하나씩 구현해 나간다.
상세 설명

비제어 흐름 명령어란, 프로그램의 실행 순서를 변경하지 않는 명령어이다. 다음 PC는 항상 PC+4가 된다.

  • R-Type ALU 명령어: ADD, SUB, AND, OR, SLT 등 — 두 레지스터 값으로 연산
  • I-Type ALU 명령어: ADDI, ANDI, ORI 등 — 레지스터와 즉시값으로 연산
  • Load/Store 명령어: LW, SW 등 — 메모리 접근

가장 단순한 R-Type부터 시작하여 점차 복잡한 명령어를 추가하며 데이터패스를 확장해 나간다.

Page 15

R-Type ALU 명령어 구현 (1)

R-Type ALU 명령어를 구현하려면 PC 증가용 가산기와 funct에 따라 동작하는 ALU가 필요하다.
상세 설명

R-Type ALU 명령어(예: ADD, SUB)의 동작을 의사 코드로 표현하면:

if type(Mem[PC]) == R-type
  {rs, rt, rd, shamt, funct} = decode(Mem[PC])
  if category(funct) == ALUs
    dest_reg ← rd
    GPR[dest_reg] ← ALU(funct, shamt, GPR[rs], GPR[rt])
PC ← PC + 4

이를 분석하면 두 가지 하드웨어가 필요하다:

  1. 가산기(Adder) #1: 다음 PC를 계산한다. 항상 PC + 4 (32비트 = 4바이트이므로).
  2. ALU: funct 필드에 따라 다양한 연산(덧셈, 뺄셈, AND, OR, SLT 등)을 수행한다.

동작 과정:

  1. PC 주소로 명령어 메모리에서 명령어를 가져온다 (IF)
  2. 명령어에서 rs, rt, rd, funct 등을 분리한다 (ID) — 사실 하드웨어에서는 "배선(wiring)"으로 간단히 구현
  3. 레지스터 파일에서 GPR[rs]와 GPR[rt] 값을 읽는다
  4. ALU가 funct에 따라 두 값을 연산한다 (EX)
  5. 결과를 GPR[rd]에 쓴다 (WB)
  6. PC를 PC + 4로 갱신한다
핵심 용어
영어한국어설명
funct 기능 코드 R-Type에서 실제 연산 종류를 결정하는 6비트 필드
dest_reg 목적지 레지스터 연산 결과가 저장될 레지스터. R-Type에서는 rd
GPR 범용 레지스터 General Purpose Register. MIPS에는 32개 존재 ($0~$31)
Page 16

R-Type ALU 명령어 구현 (2) — 데이터패스

R-Type ALU의 데이터패스를 구체적인 회로로 표현한다. 명령어 디코딩은 단순한 배선(Wiring)으로 구현된다.
데이터패스 다이어그램
+-----+ 4 -->| Add |--> PC+4 +-----+ ^ | +----+ addr +------+ rs +----------+ | PC |---+------->| Inst |---+---->| r1 *r1 |----+ +----+ | | Mem | | | | | ALU op | +------+ | rt | r2 *r2 |----+ | | |inst +---->| | | +-----+ | | | rd | r3 | +->| ALU |--> result | | +---->| wdata | +->| | | | +----------+ +-----+ | | Register File

핵심 포인트:

  • PC + 4 계산: 별도의 가산기(Add)로 상수 4를 더한다. 32비트 머신이므로 다음 명령어는 항상 4바이트 뒤에 있다.
  • ALU 연산 선택: ALU op 신호가 funct 필드에 기반하여 ALU의 연산 모드를 결정한다.
  • 디코딩 = 배선: 32비트 명령어에서 rs(25:21), rt(20:16), rd(15:11) 등을 추출하는 것은 단순히 해당 비트 위치의 선을 연결하는 것이다. 별도의 복잡한 디코더가 필요 없다.
RISC의 장점이 여기서 드러난다. 명령어 형식이 고정되어 있으므로 디코딩이 "배선(Just wiring)"으로 가능하다. CISC에서는 가변 길이 명령어 때문에 복잡한 디코더가 필요하다.
Page 17

I-Type ALU 명령어 구현 (1)

I-Type ALU 명령어를 추가하려면 ALU 입력 선택용 MUX, 목적지 레지스터 선택용 MUX, 그리고 16비트 즉시값을 32비트로 부호 확장(Sign Extension)하는 회로가 필요하다.
상세 설명

I-Type ALU 명령어(예: ADDI)의 동작을 의사 코드로 표현하면:

if type(Mem[PC]) == I-type
  {op, rs, rt, imm} = decode(Mem[PC])
  if category(op) == ALUs
    dest_reg = rt
    GPR[dest_reg] ← ALU(op, GPR[rs], sign_extend32b(imm))
PC ← PC + 4

R-Type과의 차이점:

  1. ALU의 두 번째 입력: R-Type은 GPR[rt]이지만, I-Type은 sign_extend(imm)이다. → MUX로 선택해야 한다.
  2. 목적지 레지스터: R-Type은 rd이지만, I-Type은 rt이다. → MUX로 선택해야 한다.
  3. 부호 확장(Sign Extension): 16비트 즉시값을 32비트로 확장해야 한다. 부호 비트(MSB)를 상위 16비트에 복사한다.
핵심 용어
영어한국어설명
MUX (Multiplexer) 멀티플렉서 여러 입력 중 하나를 선택하여 출력하는 회로. 선택 신호에 의해 제어됨
Sign Extension 부호 확장 작은 비트 수의 값을 큰 비트 수로 확장할 때 부호 비트를 복사하는 방법
Immediate 즉시값 명령어 내에 직접 포함된 상수값
부호 확장 예시: 16비트 값 0xFFFC(-4)를 32비트로 확장하면 0xFFFFFFFC(-4). MSB가 1이므로 상위 16비트를 모두 1로 채운다. 양수라면 상위 비트를 0으로 채운다.

I-Type ALU 명령어 구현 (2) — 데이터패스

기존 R-Type 데이터패스에 두 개의 MUX와 부호 확장기(Sign Extend)를 추가하여 I-Type ALU를 지원한다.
데이터패스 변경사항
+-----+ 4 -->| Add |--> PC+4 +-----+ ^ +----+ addr +------+ rs +----------+ | PC |---+------->| Inst |---+---->| r1 *r1 |---+ +----+ | Mem | | | | | ALU op +------+ | rt | r2 *r2 |---+ | |inst +---->| | | +-----+ | rd +--+ | r3 | +-| ALU | | +----+ | | wdata | | | | [MUX] +----------+ +-| | | (RegDst) Reg File | +-----+ | | | imm +--------+ [MUX] +-(16b)--->| Sign |--32b---+ (ALUSrc) | Extend | +--------+

추가된 요소들:

  • MUX (RegDst): 목적지 레지스터를 선택한다. I-type이면 rt를, R-type이면 rd를 r3(write register)에 연결한다.
  • MUX (ALUSrc): ALU의 두 번째 입력을 선택한다. I-type이면 부호 확장된 즉시값을, R-type이면 GPR[rt] 값을 선택한다.
  • Sign Extend: 16비트 즉시값(imm)을 32비트로 부호 확장한다. 일반 ALU 연산에서는 부호 있는(Signed) 확장이 필요하다.
MUX의 선택 신호는 "이 명령어가 I-type인가?"로 결정된다. 이 신호를 나중에 제어 유닛(Control Unit)에서 생성한다.

Load & Store 명령어 구현

Load/Store는 I-Type 형식이며, 기존 ALU를 주소 계산(Address Calculation)에 재활용하고, 데이터 메모리와 읽기/쓰기 모드 선택이 추가로 필요하다.
상세 설명

Load와 Store 명령어의 동작을 의사 코드로 표현하면:

if type(Mem[PC]) == I-type
  {op, rs, rt, imm} = decode(Mem[PC])
  address = GPR[rs] + sign_extend32b(imm)  // ALU 재활용!
  if category(op) == loads
    GPR[rt] ← Mem[address]          // 메모리에서 읽어 레지스터에 저장
  else if category(op) == stores
    Mem[address] ← GPR[rt]          // 레지스터 값을 메모리에 저장
PC ← PC + 4

핵심 관찰:

  • 주소 계산: base(GPR[rs]) + offset(sign_extend(imm))은 덧셈 연산이다. 기존 ALU를 그대로 사용하면 된다!
  • 데이터 메모리: 명령어 메모리와 별도로 데이터를 읽고 쓸 수 있는 메모리가 필요하다.
  • 읽기/쓰기 모드: Load는 메모리 읽기(MemRead), Store는 메모리 쓰기(MemWrite). 이를 선택하는 제어 신호가 필요하다.
핵심 용어
영어한국어설명
Load (LW) 적재 (워드 적재) 메모리에서 데이터를 읽어 레지스터에 저장
Store (SW) 저장 (워드 저장) 레지스터의 값을 메모리에 저장
Base + Offset 베이스 + 오프셋 메모리 주소 계산 방식: 기준 주소(rs) + 변위(imm)

Load 경로 (Load Path)

Load 명령어의 데이터 흐름을 추적한다. 주소 계산 경로(빨간색)와 데이터 로딩 경로(파란색) 두 가지로 나뉜다.
데이터패스 다이어그램
+-----+ 4 -->| Add |--> PC+4 +-----+ write_en = isLoad +----+ addr +------+ rs +----------+ | PC |---+------->| Inst |---+---->| r1 *r1 |---+ add +----+ | Mem | | | | | | +------+ | rt | r2 *r2 | | +-----+ |inst +---->| | +--| ALU |--addr--+ | rd +--+ | r3 | +--| | | | [MUX] | wdata | | +-----+ +--------+ | +----------+ [MUX] | Data | | Reg File (ALUSrc) | Memory | | | | rdata |--+ | imm +--------+ | +--------+ | +-(16b)--->| Sign |--32b---+ | | Extend | | +--------+ result ←-----------+

Load 동작 순서:

  1. 주소 계산(빨간색 경로): GPR[rs] 값과 부호 확장된 imm이 ALU에 입력되어 메모리 주소를 계산한다.
  2. 데이터 로딩(파란색 경로): 계산된 주소로 데이터 메모리에서 값을 읽어(rdata), 레지스터 파일의 rt에 기록한다.
  3. write_en = isLoad: 레지스터 파일의 쓰기 활성화 신호. Load일 때 활성화되어야 한다.
Load에서는 ALU 결과가 아닌 데이터 메모리 출력(rdata)이 레지스터에 기록된다. 따라서 "ALU 결과 vs 메모리 데이터"를 선택하는 MUX가 추가로 필요하다(다이어그램에서는 isLoad MUX로 표현).
Page 21

Store 경로 (Store Path)

Store 명령어도 Load와 동일한 주소 계산 경로를 사용하지만, 데이터 흐름의 방향이 반대이다. 레지스터 값이 데이터 메모리에 기록된다.
상세 설명

Store 동작 순서:

  1. 주소 계산(빨간색 경로): Load와 동일. GPR[rs] + sign_extend(imm)으로 메모리 주소를 계산한다.
  2. 데이터 저장(파란색 경로): GPR[rt]의 값을 읽어 데이터 메모리의 wdata 포트에 전달하고, 계산된 주소에 기록한다.
  3. write_en = isLoad: Store일 때는 레지스터 파일에 쓰지 않으므로 write_en이 비활성화(0)된다.

Store의 특징:

  • 레지스터 파일에는 쓰지 않는다 (WB 단계가 실질적으로 없음).
  • 대신 데이터 메모리에 쓴다 (MemWrite 활성화).
  • GPR[rt]는 레지스터 파일의 두 번째 읽기 포트(*r2)에서 읽혀 데이터 메모리의 쓰기 데이터(wdata)로 전달된다.
Load vs Store 헷갈리지 않기: Load는 메모리 → 레지스터 (데이터를 "적재"), Store는 레지스터 → 메모리 (데이터를 "저장"). 주소 계산 방식은 동일하다.

비제어 흐름 명령어 전체 조직(Organization)

R-Type ALU, I-Type ALU, Load, Store를 모두 지원하는 데이터패스가 완성되었다. isLoad MUX로 ALU 결과와 메모리 데이터 중 하나를 선택한다.
종합 데이터패스
+-----+ 4 -->| Add |--> PC+4 +-----+ write_en +----+ addr +------+ rs +----------+ | PC |---+------->| Inst |---+---->| r1 *r1 |---+ op +----+ | Mem | | | | | | +------+ | rt | r2 *r2 |---+ +-----+ |inst +---->| | | | ALU |--addr--+ | rd +--+ | r3 | +---| | | | [MUX] | wdata | | +-----+ +--------+ | (RegDst) +----+-----+ [MUX] | Data | | Reg File (ALUSrc) | Memory | | ^ | | rdata |-+ | imm +--------+ | +--------+ | +-(16b)--->| Sign |--32b---+ | | Extend | [MUX]←-----+ +--------+ (isLoad) 0: ALU result 1: Memory data

현재 데이터패스의 구성 요소:

  • PC: 현재 명령어 주소
  • 가산기(Add): PC + 4 계산
  • 명령어 메모리(Inst Memory): 명령어 인출
  • 레지스터 파일(Register File): 2개 읽기 포트, 1개 쓰기 포트
  • ALU: 연산 수행 또는 주소 계산
  • 데이터 메모리(Data Memory): Load/Store용
  • MUX 3개: RegDst(목적지 레지스터 선택), ALUSrc(ALU 입력 선택), isLoad(WB 데이터 선택)
  • 부호 확장기(Sign Extend): 16비트 → 32비트
이 데이터패스는 비제어 흐름 명령어만 지원한다. PC는 항상 PC+4로 갱신되므로, 점프나 분기를 할 수 없다. 다음 섹션에서 제어 흐름 명령어를 추가한다.

목차: 제어 흐름 명령어 구현

이제 프로그램의 실행 흐름을 변경하는 제어 흐름 명령어(Control Flow Instructions)를 구현한다: 무조건 점프(J, JAL, JR)와 조건 분기(BEQ, BNE 등).
상세 설명

제어 흐름 명령어는 다음 PC를 PC+4가 아닌 다른 값으로 설정하는 명령어이다:

  • 무조건 점프(Unconditional Jump): J, JAL, JR — 항상 특정 주소로 점프
  • 조건 분기(Conditional Branch): BEQ, BNE 등 — 조건이 참이면 분기, 거짓이면 PC+4 계속

이를 위해 데이터패스에 PC 선택용 MUX, 분기 주소 계산용 가산기 등이 추가되어야 한다.

Page 24

무조건 점프 구현 (1) — J, JAL, JR

무조건 점프에는 J/JAL(J-Type)과 JR(R-Type) 두 종류가 있다. 대상 주소(Target Address) 계산 방식이 다르며, 다양한 PC 값을 선택하는 MUX가 필요하다.
명령어 인코딩
J/JAL (J-Type): | opcode | immediate (26-bit) | | 6-bit | 26-bit | JR (R-Type): | 0 (fixed) | rs | rt | rd | shamt | funct | | 6-bit | 5-bit | 5-bit | 5-bit | 5-bit | 6-bit |
상세 설명 — 동작 의사 코드
if type(Mem[PC]) == J-type
  {op, imm} = decode(Mem[PC])
else if type(Mem[PC]) == R-type
  {op, rs, ...} = decode(Mem[PC])

if op is J || JAL
  target ← {(PC+4)[31:28], imm, 2'b00}  // ISA 정의!
  if op is JAL
    GPR[ra] ← PC + 4                     // 복귀 주소 저장
else if op is JR
  target ← GPR[rs]                        // 레지스터 값으로 점프

PC ← target

각 점프의 대상 주소 계산 방법:

  • J / JAL: 대상 주소 = {(PC+4)의 상위 4비트, 26비트 immediate, 00}. 즉, PC+4의 상위 4비트를 유지하고, 26비트 즉시값에 2비트(00)를 붙여 28비트를 만든 후 결합한다. 워드 정렬이므로 하위 2비트는 항상 00이다.
  • JAL 추가 동작: 복귀 주소(PC+4)를 \$ra(레지스터 31)에 저장한다. 서브루틴 호출에 사용된다.
  • JR: GPR[rs] 값을 직접 대상 주소로 사용한다. 주로 함수 복귀(jr $ra)에 사용된다.
핵심 용어
영어한국어설명
J (Jump) 점프 26비트 즉시값으로 계산된 주소로 무조건 점프
JAL (Jump and Link) 점프 및 링크 점프하면서 복귀 주소를 $ra에 저장 (함수 호출)
JR (Jump Register) 레지스터 점프 레지스터에 저장된 주소로 점프 (함수 복귀)
Concatenation 연결 비트들을 이어붙이는 연산. J/JAL의 대상 주소 계산에 사용
J/JAL의 주소 계산에서 (PC+4)[31:28]을 사용하는 이유: 26비트 즉시값에 2비트(00)를 붙여도 28비트뿐이다. 32비트 주소를 만들려면 나머지 4비트가 필요한데, 이를 PC+4의 상위 4비트에서 가져온다. 이는 ISA가 정의한 규칙이다.
Page 25

J 명령어의 데이터 경로

J 명령어를 위해 데이터패스에 연결기(Concatenator)와 PC 선택 MUX가 추가된다.
데이터패스 변경사항
+-----+ 4 -->| Add |--> PC+4 ---[31:28]--+ +-----+ | write_en +--------+ +----+ addr +------+ rs +------| Concat | | PC |---+------->| Inst |---+---->| r1 +--------+ +----+<--[MUX] | Mem | | | | ^ (PC_sel) +------+ | rt | r2 *r2 | | |inst +---->| | +--- PC+4 | rd +--+ | r3 | +--- Concat out | [MUX] | wdata | | (RegDst) +----------+ | Reg File | | imm +--------+ +-(16b)--->| Sign |--32b +-(26b)--->| Extend | +--------+

J 명령어의 실행 경로:

  1. 명령어에서 26비트 즉시값을 추출한다.
  2. Concatenator가 {(PC+4)[31:28], imm[25:0], 2'b00}을 결합하여 32비트 대상 주소를 만든다.
  3. PC 선택 MUX가 Concatenator 출력을 선택하여 다음 PC로 설정한다.

J 명령어는 레지스터에 쓰지 않으므로 write_en은 비활성화된다.

Page 26

JR 명령어의 데이터 경로

JR 명령어는 GPR[rs]의 값을 직접 다음 PC로 사용한다. 레지스터 파일의 r1(= *r1 = GPR[rs]) 출력이 PC 선택 MUX의 입력으로 연결된다.
상세 설명

JR의 데이터패스는 기존 구조를 크게 변경하지 않는다:

  • 레지스터 파일에서 GPR[rs]를 읽는 경로(*r1)는 이미 존재한다.
  • 이 *r1 출력을 PC 선택 MUX의 추가 입력으로 연결하기만 하면 된다.
  • PC 선택 MUX가 이제 최소 3가지 입력을 가진다: PC+4, Concatenator 출력(J/JAL), r1 출력(JR).

JR은 R-Type 명령어이므로 opcode가 0이고, funct 필드로 JR임을 식별한다.

JR은 주로 jr $ra로 사용되어 함수에서 복귀한다. JAL로 함수를 호출하면서 $ra에 복귀 주소를 저장하고, 함수 끝에서 JR $ra로 돌아온다.
Page 27

조건 분기(Conditional Branch) 구현

조건 분기(BEQ, BNE 등)는 I-Type 형식이며, 분기 대상 주소 계산용 전용 가산기와 오프셋 정렬용 시프터(Shifter)가 필요하다.
상세 설명

조건 분기 명령어의 동작을 의사 코드로 표현하면:

if type(Mem[PC]) == I-type
  {op, rs, rt, imm} = decode(Mem[PC])
  target ← PC + 4 + sign_extend(imm) << 2
  if op is conditional branch
    if condition(op, GPR[rs], GPR[rt]) == TRUE
      PC ← target          // Taken
    else
      PC ← PC + 4          // Not Taken

분기 대상 주소 계산의 핵심:

  • Immediate 필드의 의미: Taken 경로와 Not-Taken 경로 사이의 거리(Distance)를 명령어 단위로 나타낸다. 바이트 단위가 아니라 명령어 단위이다.
  • 왼쪽 시프트 2비트(<<2): 명령어 단위 거리를 바이트 단위로 변환한다. 4바이트 정렬이므로 x4 = <<2.
  • PC + 4 기준: 분기 대상은 현재 PC가 아닌 PC+4를 기준으로 오프셋을 더한다.
  • 전용 가산기(2nd Adder): PC+4에 시프트된 오프셋을 더하는 별도의 가산기가 필요하다. 기존 ALU는 조건 비교(GPR[rs] vs GPR[rt])에 사용되기 때문이다.
핵심 용어
영어한국어설명
Conditional Branch 조건 분기 조건이 참이면 분기, 거짓이면 다음 명령어 실행
Taken 분기 발생 조건이 참이어서 분기 대상으로 이동하는 경우
Not Taken 분기 비발생 조건이 거짓이어서 PC+4로 진행하는 경우
Branch Offset 분기 오프셋 분기 대상까지의 거리(명령어 단위). 부호 확장 후 <<2로 바이트 변환
ALU가 이중 역할: 비제어 흐름 명령어에서는 ALU가 연산/주소 계산에 쓰이지만, 조건 분기에서는 조건 비교(예: rs == rt?)에 사용된다. 분기 대상 주소는 별도 가산기로 계산해야 한다.
Page 28

조건 분기: Taken 경로

조건이 참(Taken)인 경우, 2차 가산기가 계산한 분기 대상 주소가 다음 PC로 선택된다.
데이터패스 다이어그램
+------+ +-----+ <<2 -->| Add2 |--> Branch Target 4 -->| Add |--> PC+4 ----+------------------>| | +-----+ | +------+ write_en op +----+ addr +------+ rs +----------+ | PC |---+------->| Inst |---+---->| r1 *r1 |---+ +----+<--[MUX] | Mem | | | | | +-----+ ^ (PC_sel) +------+ | rt | r2 *r2 |---+---| ALU |--addr | |inst +---->| | | | cmp | +--- PC+4 | rd +--+ | r3 | | +-----+ +--- Concat | [MUX] | wdata | [MUX] +--- r1 (JR) | +----------+ (ALUSrc) +--- BranchTarget | Reg File | | imm +--------+ | Concat. +-(16b)--->| Sign |--32b---+ +-(26b) | Extend | +--------+

Taken 경로의 동작:

  1. ALU가 GPR[rs]와 GPR[rt]를 비교한다 (예: BEQ면 뺄셈 수행 후 결과가 0인지 확인).
  2. 비교 결과가 "조건 충족"이면 Taken 신호가 활성화된다.
  3. 2차 가산기(Add2)가 PC+4 + (sign_extend(imm) << 2)를 계산한다.
  4. PC 선택 MUX가 Branch Target을 선택하여 다음 PC로 설정한다.
분기에서는 ALUSrc MUX가 GPR[rt] 값을 ALU에 전달해야 한다 (부호 확장된 즉시값이 아님). 분기의 ALU 입력은 R-Type과 동일하게 두 레지스터 값이다.
Page 29

조건 분기: Not-Taken 경로

조건이 거짓(Not-Taken)인 경우, 다음 PC는 평소처럼 PC + 4가 된다. 분기 대상 주소는 무시된다.
상세 설명

Not-Taken 경로에서는:

  1. ALU가 동일하게 조건을 비교하지만, 결과가 "조건 미충족"이다.
  2. Taken 신호가 비활성화(0)된다.
  3. PC 선택 MUX가 PC+4를 선택한다.
  4. 프로그램은 분기 없이 다음 명령어로 순차 진행한다.

2차 가산기는 여전히 분기 대상 주소를 계산하지만, 그 결과가 사용되지 않는다. 싱글사이클에서는 모든 구성 요소가 매 사이클 동작하므로, 불필요한 계산도 수행된다 (에너지 낭비).

에너지 낭비: 싱글사이클에서는 Not-Taken이든 Taken이든 모든 하드웨어가 매 사이클 동작한다. 사용하지 않는 경로의 계산도 전력을 소비한다. 이는 싱글사이클의 단점 중 하나이다.
Page 30

목차: 제어 유닛(Control Unit)

데이터패스의 모든 구성 요소가 갖춰졌다. 이제 이들을 올바르게 제어하는 제어 유닛(Control Unit)을 설계한다.
상세 설명

지금까지 만든 데이터패스에는 여러 MUX, 메모리 읽기/쓰기 활성화 신호, ALU 연산 모드 등 제어가 필요한 신호가 많다. 어떤 명령어가 실행되느냐에 따라 이 신호들이 적절하게 설정되어야 한다.

이 모든 제어 신호를 opcode(명령어의 종류)로부터 생성하는 것이 제어 유닛의 역할이다.

Page 31

구성 요소를 어떻게 제어할 것인가?

프로세서에는 MUX 선택 신호, 메모리 읽기/쓰기 활성화 신호, ALU 모드 제어 등 많은 제어 신호가 필요하다. 중앙 제어 유닛(Central Control Unit)이 opcode를 입력으로 받아 이들을 생성한다.
상세 설명

제어가 필요한 신호들을 정리하면 크게 세 가지 범주이다:

  • MUX 선택 신호(Selection Signals): RegDst, ALUSrc, MemtoReg(isLoad), PC 선택 등
  • 활성화 신호(Enable Signals): 메모리의 읽기/쓰기(MemRead, MemWrite), 레지스터의 쓰기(RegWrite) 등
  • 모드 제어 신호(Mode Control): ALU의 연산 모드(ALUOp) 등

제어 유닛의 구조는 단순하다:

  • 입력: opcode (명령어의 상위 6비트)
  • 출력: 각종 제어 신호

제어 유닛은 본질적으로 조합 논리 회로이다. opcode를 입력으로 받아, 해당 명령어에 맞는 제어 신호 조합을 즉시 출력한다.

제어 유닛 출력: 컴포넌트 제어 신호

레지스터 쓰기(RegWrite), 메모리 읽기(MemRead), 메모리 쓰기(MemWrite) 신호를 opcode로부터 결정하는 논리식을 정의한다.
상세 설명

각 제어 신호의 논리식:

  • RegWrite = (opcode != SW && opcode != BXX && opcode != J && opcode != JR)
  • 즉, Store, 분기, J, JR을 제외한 모든 명령어에서 레지스터에 결과를 기록한다. ALU 명령어와 Load가 여기 해당한다.
  • MemRead = (opcode == LW)
  • Load Word일 때만 데이터 메모리를 읽는다.
  • MemWrite = (opcode == SW)
  • Store Word일 때만 데이터 메모리에 쓴다.
JAL 지원 문제: JAL은 점프하면서 GPR[$ra]에 PC+4를 저장해야 한다. 따라서 JAL도 RegWrite가 활성화되어야 하고, 쓰기 데이터로 PC+4를, 쓰기 대상으로 $ra(31번 레지스터)를 선택하는 추가 경로가 필요하다. 이는 추가적인 MUX로 해결한다.
제어 신호 요약표
제어 신호활성화 조건역할
RegWrite SW, BXX, J, JR 제외 레지스터 파일 쓰기 활성화
MemRead opcode == LW 데이터 메모리 읽기 활성화
MemWrite opcode == SW 데이터 메모리 쓰기 활성화

제어 유닛 출력: MUX 제어 신호

RegDst와 ALUSrc 두 MUX의 선택 신호를 opcode로부터 결정한다.
상세 설명

MUX 제어 신호의 논리식:

  • RegDst = (opcode == 0)
    • 참(1)이면: rd를 쓰기 레지스터(r3)로 선택 → R-Type 명령어
    • 거짓(0)이면: rt를 쓰기 레지스터(r3)로 선택 → I-Type 명령어 (ADDI, LW 등)
  • ALUSrc = (opcode != 0 && opcode != BXX)
    • 참(1)이면: 부호 확장된 즉시값(immediate)을 ALU의 두 번째 입력으로 전달 → I-Type ALU, Load/Store의 주소 계산
    • 거짓(0)이면: GPR[rt] 값을 ALU의 두 번째 입력으로 전달 → R-Type ALU, 분기의 조건 비교
분기(BXX) 주의: 분기는 I-Type이지만 ALU에 즉시값이 아닌 GPR[rt]를 넣어야 한다(조건 비교용). 따라서 ALUSrc에서 분기를 R-Type과 같은 방향으로 처리한다. 이 점을 놓치면 분기가 제대로 동작하지 않는다.
MUX 제어 요약표
MUX선택 조건0일 때1일 때
RegDst opcode == 0 rt → r3 (I-Type) rd → r3 (R-Type)
ALUSrc opcode != 0 && opcode != BXX GPR[rt] (R-Type, Branch) sign_extend(imm) (I-Type ALU, Load/Store)

제어 유닛 출력: PC 선택

다음 PC 값은 4가지 경우 중 하나이다. PC 선택을 위해 추가 입력 신호(Jump, Taken)가 필요하다.
상세 설명

PC 선택 MUX는 4가지 입력을 가진다:

  1. Concatenator 출력: Jump == true일 때 → J 또는 JAL 명령어
    • Jump = (opcode == J || opcode == JAL)
  2. r1 (GPR[rs]): opcode == 0 && instruction[5:0] == JR일 때 → JR 명령어
  3. 2차 가산기 출력: opcode == BXX && Taken == 1일 때 → 조건 분기가 발생한 경우
  4. PC + 4: 위 세 경우에 해당하지 않을 때 → 일반적인 경우 또는 분기가 Not-Taken인 경우

PC 선택은 단순한 2비트 MUX가 아니라, 여러 조건을 조합하여 우선순위에 따라 결정된다.

PC 선택 요약표
다음 PC 값조건해당 명령어
Concat output Jump == 1 J, JAL
GPR[rs] (r1) opcode==0 && funct==JR JR
2nd Adder output opcode==BXX && Taken==1 BEQ/BNE (Taken)
PC + 4 그 외 모든 경우 ALU, Load/Store, BXX(Not-Taken)
PC 선택에는 opcode만으로는 부족하다! Jump 신호(opcode 기반)와 Taken 신호(ALU 비교 결과 기반) 두 가지 정보가 모두 필요하다. 제어 유닛이 이 두 신호를 결합하여 최종 PC 선택을 결정한다.
