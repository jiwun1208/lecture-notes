강의 소개

과목 정보

과목: System Programming (CSE306) · DGIST

주제: Chapter 4 — Machine-level Programming I: Basics

담당: Prof. Yongwoo Lee

이 강의는 C 코드가 기계어(Machine Code)로 변환되는 과정과, x86-64 어셈블리의 기초를 다룹니다.

C 언어 뒤에서 실제로 어떤 일이 일어나는지를 배웁니다. 컴파일러가 C 코드를 어떻게 어셈블리 명령어로 변환하는지, CPU의 레지스터와 메모리를 어떻게 사용하는지를 이해합니다.

구체적으로, 다음 네 가지 핵심 질문에 답할 수 있게 됩니다: (1) C의 변수는 어셈블리에서 어떻게 표현되는가? — 레지스터와 메모리 주소로 매핑됩니다. (2) C의 포인터 역참조(*p)는 어셈블리에서 어떻게 구현되는가? — 간접 주소 지정 모드 (%reg)로 표현됩니다. (3) 함수 호출은 어떻게 이루어지는가? — call과 ret 명령어, 그리고 스택과 레지스터 기반 인자 전달 규약을 사용합니다. (4) 컴파일러는 왜 곱셈 대신 shift와 leaq를 사용하는가? — 곱셈 명령어보다 훨씬 빠르기 때문입니다.

Intel x86 프로세서

Intel x86은 랩탑/데스크탑/서버 시장을 지배하는 프로세서 아키텍처입니다.

핵심 특징:
진화적 설계(Evolutionary Design): 1978년 8086부터 하위 호환을 유지하며 발전
CISC (Complex Instruction Set Computer): 다양한 명령어와 형식. 하지만 Linux에서는 일부만 사용
CISC vs RISC
CISC (x86)RISC (ARM, RISC-V)
명령어 수매우 많음적음
명령어 형식다양균일
장점속도 (Intel 최적화)저전력, 단순 설계
현재 RISC 르네상스: ARM(모바일), RISC-V(오픈소스)가 저전력 시장에서 부상 중입니다.

재미있는 사실: 현대 x86 프로세서는 내부적으로 CISC 명령어를 micro-ops(RISC 스타일의 작은 연산)로 분해하여 실행합니다. 즉, 외부는 CISC이지만 내부 실행 엔진은 사실상 RISC입니다. Intel은 이 방식으로 복잡한 명령어 세트의 호환성을 유지하면서도 파이프라인 효율을 극대화합니다.

어셈블리/기계어 관점 — CPU 상태

어셈블리 프로그래머가 볼 수 있는 프로세서 상태(Programmer-Visible State):

구성 요소설명
PC (Program Counter)다음 실행할 명령어의 주소. x86-64에서는 %rip이라 부름
Register File16개의 범용 레지스터. 자주 사용되는 프로그램 데이터 저장
Condition Codes가장 최근 산술/논리 연산의 상태 정보. 조건 분기에 사용
Memory바이트 단위로 주소 지정. 코드, 데이터, 스택 저장
Figure — CPU Programmer-Visible State 다이어그램
CPU 내부 상태 구조:
[PC (%rip)] ──→ 다음 실행할 명령어의 메모리 주소를 가리킴
[Register File] ──→ 16개 범용 레지스터 (%rax, %rbx, ..., %r15), 각 64비트
[Condition Codes] ──→ CF(Carry), ZF(Zero), SF(Sign), OF(Overflow) 플래그
[Memory] ──→ 가상 주소 공간: Code(텍스트) | Data(전역) | Heap(동적) | Stack(함수)

실행 흐름: PC가 가리키는 주소에서 명령어를 fetch → 명령어 decode → 레지스터/메모리에서 operand 읽기 → ALU 실행 → 결과를 레지스터/메모리에 write → PC 업데이트 (다음 명령어 또는 분기 대상)

Condition Codes는 addq, subq 등의 산술 연산 후 자동으로 설정됩니다. 예를 들어 subq %rax, %rbx 실행 후 결과가 0이면 ZF=1, 음수이면 SF=1이 됩니다. 이 플래그들은 je(ZF=1이면 점프), jl(SF!=OF이면 점프) 등의 조건 분기에서 사용됩니다.

C → 실행 파일 변환 과정

gcc -Og p1.c p2.c -o p

변환은 4단계로 이루어집니다:

C program (p1.c, p2.c)        [텍스트]
        ↓  Compiler (gcc -Og -S)
Asm program (p1.s, p2.s)       [텍스트]
        ↓  Assembler (as)
Object program (p1.o, p2.o)    [바이너리]
        ↓  Linker (ld)         + Static libraries (.a)
Executable program (p)         [바이너리]
-Og: 디버깅에 적합한 기본 최적화. 어셈블리를 읽기 쉽게 만듦. -O2나 -O3는 적극적 최적화로 읽기 어려운 코드 생성.
Figure — 컴파일 파이프라인 상세
[p1.c, p2.c] (C 소스, 텍스트)
  │ Preprocessor (cpp): #include 확장, #define 매크로 치환, 조건부 컴파일 처리
  ▼
[p1.i, p2.i] (전처리된 C 소스, 텍스트)
  │ Compiler (cc1): 구문 분석 → AST → 중간 표현(IR) → 최적화 → 코드 생성
  ▼
[p1.s, p2.s] (어셈블리, 텍스트) — 사람이 읽을 수 있는 마지막 단계
  │ Assembler (as): 니모닉을 바이너리 인코딩으로 변환. 심볼 테이블 생성
  ▼
[p1.o, p2.o] (오브젝트 파일, 바이너리) — ELF 형식. 재배치 가능(relocatable)
  │ Linker (ld): 심볼 해결, 주소 재배치, 라이브러리(.a/.so) 결합
  ▼
[p] (실행 파일, 바이너리) — 최종 ELF 실행 파일. 절대 주소 확정

각 단계를 개별 확인하는 명령어:
gcc -E sum.c -o sum.i (전처리만) → gcc -S sum.c -o sum.s (어셈블리까지) → gcc -c sum.c -o sum.o (오브젝트까지) → gcc sum.o -o sum (링킹)

C에서 어셈블리로 — 컴파일 결과

C Code (sum.c)
long plus(long x, long y);
void sumstore(long x, long y, long *dest) {
    long t = plus(x, y);
    *dest = t;
}

gcc -Og -S sum.c → sum.s 생성:

Assembly (sum.s)
sumstore:
    pushq   %rbx          # rbx를 스택에 저장 (callee-save)
    movq    %rdx, %rbx    # dest를 rbx에 복사
    call    plus           # plus(x, y) 호출 → 결과는 %rax에
    movq    %rax, (%rbx)   # *dest = t (rax의 값을 rbx가 가리키는 메모리에)
    popq    %rbx           # rbx 복원
    ret                    # 반환
C의 6줄 함수가 어셈블리 6줄 명령어로 변환되었습니다. 각 줄이 CPU가 실행하는 하나의 동작에 대응합니다.

System V AMD64 ABI 호출 규약에 따른 레지스터 매핑을 확인합시다:

  • %rdi = 첫 번째 인자 x
  • %rsi = 두 번째 인자 y
  • %rdx = 세 번째 인자 *dest (포인터)
  • %rax = 함수 반환값 (plus의 결과 t)

pushq %rbx / popq %rbx가 필요한 이유: %rbx는 callee-saved 레지스터입니다. sumstore가 %rbx를 사용하기 전에 원래 값을 스택에 백업하고, 함수 끝에서 복원해야 합니다. 만약 이를 하지 않으면 sumstore를 호출한 함수의 %rbx 값이 손상됩니다. %rdx(dest)를 %rbx에 복사하는 이유는, call plus가 %rdx를 덮어쓸 수 있기 때문입니다 (%rdx는 caller-saved).

어셈블리의 데이터 타입

어셈블리는 타입이 없는(untyped) 언어입니다.

데이터:
"정수" 데이터: 1, 2, 4, 8 바이트 — 값 또는 주소(포인터)
부동소수점 데이터: 4, 8, 10 바이트
배열이나 구조체 같은 집합 타입은 없음 — 메모리에 연속 배치된 바이트일 뿐

코드: 명령어를 인코딩한 바이트 시퀀스

컴파일러가 타입 정보를 모두 없앤다는 뜻입니다. 어셈블리 수준에서는 int와 포인터가 구분되지 않습니다.

Object Code — 기계어

어셈블러(Assembler)가 .s 파일을 .o 파일(바이너리)로 변환합니다:

sumstore 기계어 인코딩
sumstore 시작 주소: 0x0400595

pushq  %rbx         →  0x53             (1 바이트)
movq   %rdx, %rbx   →  0x48 0x89 0xd3   (3 바이트)
call   plus          →  0xe8 0xf2 0xff 0xff 0xff  (5 바이트)
movq   %rax, (%rbx)  →  0x48 0x89 0x03   (3 바이트)
popq   %rbx          →  0x5b             (1 바이트)
ret                  →  0xc3             (1 바이트)
                                 총 14 바이트
각 명령어는 1, 3, 또는 5 바이트로 인코딩됩니다. CISC의 특징 — 명령어마다 길이가 다릅니다. (RISC는 보통 고정 4바이트)
Figure — 명령어 인코딩 구조 상세
x86-64 명령어 바이트 분해:

pushq %rbx → 0x53 (1바이트)
  0x50 = push 기본 opcode + 0x03 = %rbx 레지스터 번호. push/pop은 레지스터 번호가 opcode에 내장되어 1바이트로 인코딩됩니다.

movq %rdx, %rbx → 0x48 0x89 0xd3 (3바이트)
  0x48 = REX.W prefix (64비트 오퍼랜드 크기 지정)
  0x89 = MOV r/m64, r64 opcode
  0xd3 = ModR/M 바이트 (mod=11 레지스터→레지스터, reg=010 %rdx, rm=011 %rbx)

call plus → 0xe8 0xf2 0xff 0xff 0xff (5바이트)
  0xe8 = CALL rel32 opcode
  0xfffffff2 = -14 (little-endian). 현재 PC 기준 상대 오프셋으로 plus 함수의 위치를 지정

가변 길이 인코딩의 장점: 자주 쓰는 명령어(push, pop, ret)는 1바이트로 짧게 인코딩되어 코드 크기가 작아집니다. 단점: CPU가 명령어 경계를 찾기 어려워 디코딩이 복잡합니다.

링킹 — Linker

링커(Linker)의 역할:

  • 여러 .o 파일 간의 참조를 해결 (예: sumstore에서 plus 호출)
  • 정적 라이브러리(.a)와 결합 (malloc, printf 등의 코드)
  • 일부 라이브러리는 동적 링킹 — 프로그램 실행 시점에 연결

정적 링킹 vs 동적 링킹:

  • 정적 링킹(.a): 라이브러리 코드가 실행 파일에 복사됨. 실행 파일 크기가 커지지만 독립적으로 실행 가능.
  • 동적 링킹(.so/.dll): 실행 시점에 공유 라이브러리를 메모리에 매핑. 여러 프로그램이 같은 라이브러리를 공유하여 메모리 절약. ldd 명령어로 실행 파일이 의존하는 공유 라이브러리를 확인할 수 있습니다.

링커가 해결하는 구체적 문제: sumstore에서 call plus를 실행할 때, 어셈블러 단계에서는 plus 함수의 최종 주소를 알 수 없습니다 (다른 .o 파일에 있으므로). 링커가 모든 .o 파일을 결합하면서 plus의 실제 주소를 계산하고, call 명령어의 상대 오프셋을 패치합니다.

함수 인자 전달 (caller-saved):
  %rdi = 1번째 인자 | %rsi = 2번째 | %rdx = 3번째 | %rcx = 4번째 | %r8 = 5번째 | %r9 = 6번째

Caller-saved (호출자가 백업 책임):
  %rax, %rcx, %rdx, %rsi, %rdi, %r8, %r9, %r10, %r11

Memory → Memory 직접 전송은 불가! 반드시 레지스터를 거쳐야 합니다. 이것이 CISC인 x86에서도 지키는 제약입니다.
Figure — movq 데이터 이동 방향 다이어그램

movq (%rax), %rcx   # Memory → Register (load)

Page 24

메모리 주소 지정 모드

Normal: (R) → Mem[Reg[R]]

movq (%rcx), %rax    # rcx가 가리키는 주소에서 읽기 = *rcx

Displacement: D(R) → Mem[Reg[R] + D]

movq 8(%rbp), %rdx   # rbp + 8 위치에서 읽기 = *(rbp+8)

(R)은 C의 *ptr, D(R)은 *(ptr + offset)에 해당합니다. 구조체 멤버 접근이나 스택 변수 접근에 사용됩니다.

Figure — 메모리 주소 지정 모드의 실제 사용 예
Normal (R) — 포인터 역참조:
  C: int val = *ptr;
  ASM: movq (%rdi), %rax — %rdi가 가리키는 메모리 주소에서 8바이트를 읽어 %rax에 저장
  %rdi = 0x7fff1000이면 → M[0x7fff1000]의 값을 읽음

Displacement D(R) — 구조체 멤버 접근:
  C: struct { long a; long b; } *s; ... s->b;
  ASM: movq 8(%rdi), %rax — %rdi(구조체 시작 주소)에서 8바이트 오프셋의 멤버 b를 읽음
  %rdi = 0x7fff1000이면 → M[0x7fff1000 + 8] = M[0x7fff1008]의 값을 읽음

Displacement D(R) — 스택 변수 접근:
  C: 지역 변수가 스택에 할당된 경우
  ASM: movq -16(%rbp), %rax — 프레임 포인터(%rbp) 기준 -16 오프셋의 지역 변수 읽기
  %rbp = 0x7fff2000이면 → M[0x7fff2000 - 16] = M[0x7fff1ff0]의 값을 읽음

C Code
void swap(long *xp, long *yp) {
    long t0 = *xp;
    long t1 = *yp;
    *xp = t1;
    *yp = t0;
}
Assembly
swap:
    movq  (%rdi), %rax    # t0 = *xp     (rdi = xp)
    movq  (%rsi), %rdx    # t1 = *yp     (rsi = yp)
    movq  %rdx, (%rdi)    # *xp = t1
    movq  %rax, (%rsi)    # *yp = t0
    ret
레지스터 매핑
레지스터용도
%rdi첫 번째 인자 xp
%rsi두 번째 인자 yp
%rax임시 변수 t0
%rdx임시 변수 t1

swap 실행 추적

xp = 0x120 (값 123), yp = 0x100 (값 456)일 때 단계별 실행:

단계명령어%rax%rdxM[0x120]M[0x100]
초기———123456
1movq (%rdi), %rax123—123456
2movq (%rsi), %rdx123456123456
3movq %rdx, (%rdi)123456456456
4movq %rax, (%rsi)123456456123
레지스터를 임시 저장소로 사용하여 두 메모리 값을 교환합니다. C에서 temp 변수를 쓰는 것과 같은 원리입니다.
Figure — swap 실행 시 메모리와 레지스터 상태 변화
초기 상태: %rdi = 0x120 (xp), %rsi = 0x100 (yp)
  M[0x120] = 123, M[0x100] = 456

Step 1: movq (%rdi), %rax
  M[0x120]에서 값 123을 읽어 %rax에 저장
  [Registers] %rax = 123 | [Memory] 변화 없음

Step 2: movq (%rsi), %rdx
  M[0x100]에서 값 456을 읽어 %rdx에 저장
  [Registers] %rax = 123, %rdx = 456 | [Memory] 변화 없음

Step 3: movq %rdx, (%rdi)
  %rdx의 값 456을 M[0x120]에 기록 (store)
  [Memory] M[0x120] = 456 ← 변경됨!

Step 4: movq %rax, (%rsi)
  %rax의 값 123을 M[0x100]에 기록 (store)
  [Memory] M[0x100] = 123 ← 변경됨!

최종: M[0x120] = 456, M[0x100] = 123 → 교환 완료!
핵심 포인트: Memory→Memory 직접 이동이 불가능하므로 반드시 레지스터(%rax, %rdx)를 경유해야 합니다.

완전한 메모리 주소 지정 모드

가장 일반적인 형태:

\(D(Rb, Ri, S) = Mem[Reg[Rb] + S \times Reg[Ri] + D]\)
구성 요소의미제약
D상수 변위(displacement)1, 2, 4 바이트
RbBase 레지스터16개 중 아무거나
RiIndex 레지스터%rsp 제외
SScale (배율)1, 2, 4, 8만 가능

Scale이 1, 2, 4, 8인 이유: 각각 char, short, int, long 배열의 원소 크기!

배열 접근 패턴: A[i] → M[&A + S*i] → 어셈블리에서 (%rbx,%rcx,S) 형태로 표현
Figure — 주소 지정 모드와 C 코드의 매핑
각 주소 지정 모드가 C에서 어떤 패턴에 대응하는지:

(Rb) = M[Reg[Rb]]
  C 패턴: *p (단순 포인터 역참조)
  예: movq (%rax), %rdx → val = *p;

D(Rb) = M[Reg[Rb] + D]
  C 패턴: p->field 또는 array[constant]
  예: movq 16(%rdi), %rax → val = s->third_field; (세 번째 long 멤버)

(Rb,Ri) = M[Reg[Rb] + Reg[Ri]]
  C 패턴: *(base + offset) — 가변 오프셋
  예: movq (%rdi,%rsi), %rax → val = *(char_array + index);

D(Rb,Ri,S) = M[Reg[Rb] + S*Reg[Ri] + D]
  C 패턴: a[i].field — 구조체 배열의 멤버 접근
  예: movq 8(%rdi,%rsi,16), %rax → 크기 16바이트인 구조체 배열에서 i번째 원소의 오프셋 8 멤버 접근

Scale S 값과 데이터 타입:
  S=1: char/byte 배열 | S=2: short 배열 | S=4: int/float 배열 | S=8: long/double/포인터 배열

leaq — 주소 계산 명령어

leaq Src, Dst — "Load Effective Address". 메모리 접근 없이 주소 계산 결과만 레지스터에 저장합니다.

용도 1: 포인터 산술 — p = &x[i];

용도 2: \(x + k \times y\) 형태의 산술 계산 (k = 1, 2, 4, 8)

예시: x * 12
// C Code
long m12(long x) { return x*12; }

// Assembly
leaq (%rdi,%rdi,2), %rax    # rax = x + x*2 = 3x
salq $2, %rax                # rax = 3x << 2 = 3x * 4 = 12x
leaq는 메모리를 읽지 않습니다! 주소 계산 회로를 "빌려서" 빠른 곱셈(×3, ×5, ×9 등)에 활용하는 것입니다.
Figure — leaq vs movq 동작 비교
%rdi = 0x100일 때 두 명령어의 차이:

movq (%rdi), %rax
  1. %rdi의 값 0x100을 주소로 사용
  2. 메모리 M[0x100]에 접근하여 값을 읽음
  3. 읽은 값을 %rax에 저장
  → %rax = M[0x100]의 내용 (메모리 접근 발생!)

leaq (%rdi), %rax
  1. 주소 계산: 0x100
  2. 계산된 주소 자체를 %rax에 저장 (메모리 접근 없음!)
  → %rax = 0x100 (주소 값 자체)

컴파일러의 leaq 활용 패턴 (산술 계산):
  leaq (%rdi,%rdi,2), %rax → %rax = %rdi + %rdi*2 = 3 * %rdi (×3 곱셈)
  leaq (%rdi,%rdi,4), %rax → %rax = %rdi + %rdi*4 = 5 * %rdi (×5 곱셈)
  leaq (%rdi,%rdi,8), %rax → %rax = %rdi + %rdi*8 = 9 * %rdi (×9 곱셈)
  leaq 7(%rdi,%rsi,4), %rax → %rax = %rdi + 4*%rsi + 7 (복합 계산)

leaq가 빠른 이유: CPU의 AGU(Address Generation Unit)를 사용하므로 ALU와 병렬로 실행 가능하고, 조건 코드(CF, ZF 등)를 변경하지 않습니다.

인자 순서 주의! AT&T 문법에서 결과는 두 번째 오퍼랜드에 저장됩니다. Intel 문법과 반대!
signed/unsigned 구분 없음 — Two's Complement 덕분에 비트 연산이 동일합니다.

AT&T vs Intel 문법 비교:

  • AT&T (GCC 기본): addq %rsi, %rdi → %rdi = %rdi + %rsi (Src, Dest 순서)
  • Intel (NASM, Windows): add rdi, rsi → rdi = rdi + rsi (Dest, Src 순서, % 없음)

산술 시프트(SAR) vs 논리 시프트(SHR) 차이: 오른쪽 시프트에서만 차이가 있습니다. SAR은 부호 비트를 유지하며 복사(음수 유지), SHR은 항상 0을 채움. 예를 들어 -8(0xFFF...F8)을 1비트 오른쪽 시프트하면: SAR → -4(0xFFF...FC), SHR → 매우 큰 양수(0x7FF...FC).

decq DestDest = Dest - 1
negq DestDest = -Dest (부호 반전)
notq DestDest = ~Dest (비트 반전)

negq vs notq의 차이: negq는 2의 보수 부호 반전 (-x = ~x + 1)이고, notq는 비트 단위 반전(1의 보수)입니다. 예를 들어 %rax = 5 (0x0000...0005)일 때:

  • negq %rax → %rax = -5 (0xFFFF...FFFB)
  • notq %rax → %rax = -6 (0xFFFF...FFFA) — 모든 비트 반전

incq와 decq는 addq $1, subq $1과 결과는 같지만, 인코딩이 더 짧고(1~3바이트 vs 3~4바이트) CF(carry flag)를 변경하지 않는다는 차이가 있습니다.

산술 표현식 예제 — C와 어셈블리 대응

C Code
long arith(long x, long y, long z) {
    long t1 = x + y;
    long t2 = z + t1;
    long t3 = x + 4;
    long t4 = y * 48;
    long t5 = t3 + t4;
    long rval = t2 * t5;
    return rval;
}
Assembly
arith:
  leaq  (%rdi,%rsi), %rax       # rax = x+y       (t1)
  addq  %rdx, %rax              # rax = z+t1       (t2)
  leaq  (%rsi,%rsi,2), %rdx     # rdx = y+y*2 = 3y
  salq  $4, %rdx                # rdx = 3y*16 = 48y (t4)
  leaq  4(%rdi,%rdx), %rcx      # rcx = x+4+48y    (t5)
  imulq %rcx, %rax              # rax = t2*t5       (rval)
  ret
C에는 곱셈이 2개(y*48, t2*t5)이지만, 어셈블리에는 imulq가 1개뿐입니다. y*48은 leaq + salq로 대체되었습니다:
48 = 3 × 16 → leaq로 ×3, salq $4로 ×16
Figure — 산술 표현식의 레지스터 추적
인자 매핑: %rdi = x, %rsi = y, %rdx = z

Step 1: leaq (%rdi,%rsi), %rax
  %rax = x + y = t1
  레지스터 상태: %rax=t1, %rdi=x, %rsi=y, %rdx=z

Step 2: addq %rdx, %rax
  %rax = %rax + z = t1 + z = t2
  레지스터 상태: %rax=t2, %rdi=x, %rsi=y, %rdx=z

Step 3: leaq (%rsi,%rsi,2), %rdx
  %rdx = y + 2*y = 3y (주의: 원래 %rdx에 있던 z는 더 이상 필요 없으므로 덮어씀)
  레지스터 상태: %rax=t2, %rdi=x, %rsi=y, %rdx=3y

Step 4: salq $4, %rdx
  %rdx = 3y << 4 = 3y * 16 = 48y = t4
  레지스터 상태: %rax=t2, %rdi=x, %rdx=t4

Step 5: leaq 4(%rdi,%rdx), %rcx
  %rcx = x + 48y + 4 = (x+4) + t4 = t3 + t4 = t5
  컴파일러가 t3 = x+4를 별도로 계산하지 않고 이 leaq에 통합! (최적화)

Step 6: imulq %rcx, %rax
  %rax = t2 * t5 = rval
  ret에 의해 %rax가 반환값으로 사용됨

컴파일러 최적화 포인트: 6개의 C 연산을 6개의 어셈블리 명령어로 1:1 매핑하면서도, 곱셈 하나를 leaq+salq로 대체하고, 덧셈 두 개를 하나의 leaq로 합쳤습니다.
