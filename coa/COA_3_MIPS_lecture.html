강의 소개 및 참고 자료

이 강의는 MIPS ISA(Instruction Set Architecture)를 중심으로 명령어 집합 구조를 학습한다. 교재 Chapter 2(계속)와 Appendix A.10을 참고한다.
강의 개요

ISA(명령어 집합 구조)란 프로그래머(또는 컴파일러)가 하드웨어에게 "어떤 일을 해라"라고 지시하는 방법을 정의한 규약이다. 쉽게 말해, CPU가 이해할 수 있는 명령어의 언어 규칙이다.

이 강의에서는 대표적인 RISC 프로세서인 MIPS의 명령어 형식, ALU 명령어, 데이터 이동 명령어, 제어 흐름 명령어, 그리고 함수 호출 메커니즘을 다룬다. 마지막에는 다른 ISA(RISC-V, x86)와의 비교도 포함된다.

ISA에서 정의해야 할 것들

ISA를 설계하려면 데이터 타입, 아키텍처 상태(레지스터, 메모리 등), 연산(명령어) 세 가지를 결정해야 한다.
상세 설명

ISA는 하드웨어와 소프트웨어 사이의 계약서와 같다. 이 계약서에는 다음 세 가지가 명시되어야 한다:

  • 데이터 타입(Data Type): 어떤 종류의 데이터를 다룰 것인가?
    • 정수(Integer): 부호 있는지(Signed)? 크기(Size)는?
    • 부동소수점(Floating Point)
  • 아키텍처 상태(Architectural State): 프로그래머에게 보이는 상태란?
    • 레지스터(Registers): CPU 내부의 초고속 저장 공간
    • 프로그램 카운터(Program Counter, PC): 다음에 실행할 명령어의 주소
    • 메모리(Memory): 데이터와 코드가 저장되는 공간
  • 연산(Operations): 어떤 명령어를 제공할 것인가?
    • Add, Subtract, Multiply, Move 등
핵심 용어
영어한국어설명
ISA명령어 집합 구조하드웨어가 지원하는 명령어와 그 규칙의 총체
Architectural State아키텍처 상태프로그래머가 볼 수 있는 CPU의 상태(레지스터, PC, 메모리)
Program Counter프로그램 카운터다음에 실행할 명령어의 주소를 가리키는 레지스터
ISA는 "무엇(What)"을 정의하고, 마이크로아키텍처는 "어떻게(How)" 구현할지를 정의한다. 같은 ISA라도 마이크로아키텍처는 다를 수 있다 (예: 같은 MIPS ISA를 쓰더라도 파이프라인 단계 수가 다를 수 있음).

RISC vs. CISC 복습

ISA는 크게 RISC(Reduced Instruction Set Computer)와 CISC(Complex Instruction Set Computer)로 나뉜다. 이 수업은 주로 RISC를 다루며, CISC는 간단히 소개한다.
RISC (Reduced Instruction Set Computer)

장점 (Pros):

  • 디코더 설계 복잡도가 낮다 — 명령어가 단순하고 고정 길이이므로 하드웨어 디코더가 간단해진다.
  • 컴파일러 최적화가 쉽다 — 명령어가 하드웨어의 세부 사항을 드러내므로(expose), 컴파일러가 이를 활용하여 최적화할 수 있다.

단점 (Cons):

  • 코드 팽창(Code Bloating) — 하나의 복잡한 작업을 여러 개의 단순 명령어로 나눠야 하므로, 프로그램 크기가 커진다.
CISC (Complex Instruction Set Computer)

장점 (Pros):

  • 압축적인 코드 크기 — 복잡한 명령어 하나로 많은 작업을 수행하므로, 프로그램이 짧아진다.

단점 (Cons):

  • 컴파일러 최적화가 어렵다 — 명령어 자체가 세부 동작을 숨기므로(hide), 컴파일러가 최적화하기 까다롭다.
  • 디코더 설계 복잡도가 높다 — 명령어가 가변 길이이고 종류가 다양하여 하드웨어가 복잡해진다.
이 수업의 초점: RISC를 중심으로 학습하고, CISC는 간략히 소개한다. MIPS는 대표적인 RISC 아키텍처이다.

MIPS 소개

MIPS는 Stanford 대학의 John L. Hennessy가 이끈 프로젝트에서 시작되었으며, 최초의 파이프라인 마이크로아키텍처와 명령어/데이터 캐시를 도입한 역사적인 프로세서이다.
상세 설명

MIPS(Microprocessor without Interlocked Pipelined Stages)는 RISC 아키텍처의 대표 주자이다.

  • 출발: Stanford 대학교에서 John L. Hennessy 교수가 주도한 연구 프로젝트로 시작
  • 최초 상용 제품: MIPS Computer System의 R2000 (1985년)
    • 같은 해 출시된 Intel 80386(CISC)과 경쟁 — CISC vs. RISC 구도의 상징적 사건
    • 최초의 파이프라인(Pipelined) 마이크로아키텍처
    • 최초로 명령어 캐시(Instruction Cache)와 데이터 캐시(Data Cache)를 채택한 마이크로아키텍처
  • 활용 분야: Nintendo 64, PlayStation, 네트워크 장비 등 다양한 제품에 MIPS 기반 프로세서가 사용됨
MIPS는 컴퓨터 구조 교육에서 가장 많이 사용되는 ISA이다. 교과서 "Computer Organization and Design" (Patterson & Hennessy)의 주요 예제 아키텍처이기도 하다.

MIPS 명령어 형식 (1) — 기본 필드 구조

MIPS의 명령어는 32비트 고정 길이이며, 6개의 필드로 나뉜다. 모든 명령어가 항상 6개 필드를 전부 사용하는 것은 아니다.
32비트 명령어의 6개 필드

MIPS 명령어의 기본 구조를 정적으로(statically) 바라보면 다음 6개 필드로 구성된다:

필드비트 수설명
op (opcode)6비트연산 종류를 나타내는 코드 (명령어의 "종류 번호")
rs5비트첫 번째 소스 오퍼랜드 레지스터
rt5비트두 번째 소스 오퍼랜드 레지스터 (때로는 목적지로도 사용)
rd5비트목적지 레지스터 — 연산 결과를 저장할 레지스터
shamt5비트시프트 양(Shift Amount) — 시프트 연산에서만 사용
funct6비트기능 코드(Function Code) — opcode의 보조 코드, 세부 연산을 지정

총 6 + 5 + 5 + 5 + 5 + 6 = 32비트이다.

왜 항상 6개 필드를 다 쓰지 않을까?

명령어에 따라 레지스터가 3개 필요하지 않을 수 있다. 예를 들어, 레지스터 2개와 상수 값 하나만 있으면 충분한 경우도 있다. 따라서 필드를 유연하게 재배치할 필요가 있다 — 이것이 다음 페이지에서 설명할 R/I/J 타입 구분의 이유이다.

MIPS 명령어 형식 (2) — R, I, J 타입

MIPS 명령어는 R-type, I-type, J-type 세 가지 형식으로 나뉜다. 이 구분은 연산 종류가 아니라 필드 배치 방식에 따른 분류이다.
세 가지 명령어 형식

R-type (Register type): 레지스터 3개를 사용하는 명령어

| 0 (고정) | rs (5) | rt (5) | rd (5) | shamt (5) | funct (6) |
   6비트     5비트    5비트    5비트     5비트       6비트

opcode가 항상 0이므로, 실제 연산 종류는 funct 필드로 구분한다.

I-type (Immediate type): 레지스터 2개 + 상수(Immediate) 1개

| opcode (6) | rs (5) | rt (5) | immediate (16) |
    6비트      5비트    5비트       16비트

16비트 상수를 직접 명령어 안에 담을 수 있다. 상수를 사용하는 산술 연산, 메모리 접근, 조건 분기 등에 쓰인다.

J-type (Jump type): 상수(주소) 1개만 사용

| opcode (6) | immediate (26) |
    6비트          26비트

26비트의 큰 즉시값으로 점프할 주소를 지정한다. 무조건 점프(J)나 함수 호출(JAL)에 사용된다.

주의: R/I/J 타입은 연산 종류를 분류하는 것이 아니다. 같은 카테고리의 연산(예: ALU)이라도 R-type일 수도 있고 I-type일 수도 있다. 이 분류는 순수하게 필드 배치(비트 할당) 기준이다.

카테고리 1: ALU 명령어 (1) — 개요

ALU(Arithmetic Logic Unit) 명령어는 산술 연산과 논리 연산을 수행한다. R-type과 I-type 형식을 모두 사용한다.
상세 설명

ALU 명령어는 두 가지 형식을 사용한다:

  • R-type: 레지스터 3개를 사용 — add rd, rs, rt처럼 두 소스 레지스터의 값을 연산하여 목적지 레지스터에 저장
  • I-type: 레지스터 2개 + 즉시값 — addi rt, rs, imm처럼 레지스터 하나와 상수를 연산하여 결과를 레지스터에 저장

산술 연산(add, sub 등)과 논리 연산(and, or 등) 모두 이 카테고리에 속한다.

사용하는 명령어 형식
R-type: | 0 (고정) | rs | rt | rd | shamt | funct |
I-type: | opcode   | rs | rt | immediate (16-bit) |

카테고리 1: ALU 명령어 (2) — add와 addi 예제

add는 R-type으로 레지스터 두 개를 더하고, addi는 I-type으로 레지스터와 상수를 더한다. addi의 즉시값은 부호 있는 수이므로 subi 명령어가 따로 없다.
예제 1: add rd, rs, rt (R-type)
| opcode(0) | rs | rt | rd | shamt(00000) | funct:ADD(100000) |

동작: GPR[rd] ← GPR[rs] + GPR[rt]

두 소스 레지스터(rs, rt)의 값을 더해서 목적지 레지스터(rd)에 저장한다.

  • opcode는 항상 0 (R-type의 특징)
  • funct 필드의 값 100000이 이 명령어가 ADD임을 알려준다
  • ISA가 오버플로우 예외(overflow exception)를 정의한다 — 결과가 표현 범위를 넘으면 시스템에 알림
예제 2: addi rt, rs, 0xCAFE (I-type)
| opcode(001000) | rs | rt | immediate(0xCAFE) |

동작: GPR[rt] ← GPR[rs] + 0xCAFE

  • 즉시값(Immediate)은 부호 있는 수(signed number)이다
  • 따라서 subi 명령어가 별도로 필요 없다 — 음수 즉시값을 사용하면 빼기가 된다 (예: addi rt, rs, -5)
  • 16비트 즉시값의 표현 범위: \(-2^{15}\) ~ \(2^{16}-1\)
MIPS 설계 철학: "자주 쓰는 연산을 빠르게 만들자(Make the common case fast)." 뺄셈 전용 즉시값 명령어를 만들지 않고 addi에 음수를 넣어 해결하면 명령어 수를 줄여 디코더를 단순하게 유지할 수 있다.

카테고리 1: ALU 명령어 (3) — ISA 매뉴얼

ALU 명령어의 전체 상세 정보는 MIPS IV Instruction Set 공식 매뉴얼에서 확인할 수 있으며, 마이크로아키텍처 설계 시 이 매뉴얼을 반드시 따라야 한다.
상세 설명

슬라이드에서는 ISA 매뉴얼의 실제 페이지를 보여준다. 매뉴얼에는 각 명령어에 대해 다음 정보가 포함되어 있다:

  • 명령어의 인코딩(비트 배치)
  • 동작(Operation)의 정확한 정의
  • 예외(Exception) 발생 조건
  • 명령어의 형식(Format)과 변종(Variants)

마이크로아키텍처(실제 하드웨어)를 설계할 때는 이 매뉴얼의 세부 사항을 반드시 준수해야 한다. ISA 매뉴얼은 하드웨어 설계자와 소프트웨어 개발자 모두의 "계약서"이기 때문이다.

참고: C. Price, "MIPS IV Instruction Set," 1995.

실습: 산술 연산 프로그램 컴파일

C 코드 f = (g + h) - (i + j)를 MIPS 어셈블리로 변환하는 과정을 살펴본다.
문제 설정

C 코드:

f = (g + h) - (i + j);

레지스터 배정:

  • f, g, h, i, j → r0, r1, r2, r3, r4
  • 임시 레지스터: t0
MIPS 어셈블리 코드
ADD t0, r1, r2    // t0 = g + h
ADD r0, r3, r4    // f = i + j  (임시로 f에 저장)
SUB r0, t0, r0    // f = t0 - f = (g+h) - (i+j)

설명:

  1. 먼저 g + h를 계산하여 임시 레지스터 t0에 저장
  2. 다음 i + j를 계산하여 r0(f)에 임시 저장
  3. 마지막으로 t0 - r0, 즉 (g+h) - (i+j)를 계산하여 r0(f)에 최종 저장
MIPS의 ALU 명령어는 한 번에 오퍼랜드 2개만 처리할 수 있다. 따라서 복잡한 수식은 여러 명령어로 분해해야 하며, 임시 레지스터(temporary register)가 필요하다. 이것이 RISC의 "코드 팽창" 특성이다.

카테고리 2: 데이터 이동 명령어 (1) — 개요

데이터 이동 명령어는 메모리와 레지스터 사이에서 데이터를 옮기는 명령어이다. I-type 형식을 사용한다.
상세 설명

ALU 명령어가 연산을 수행하려면, 먼저 데이터가 레지스터에 있어야 한다. 하지만 대부분의 데이터는 메모리에 저장되어 있다. 따라서:

  1. Load: 메모리에서 레지스터로 데이터를 가져온다
  2. ALU 명령어로 연산 수행
  3. Store: 결과를 레지스터에서 메모리로 다시 저장한다

이것을 Load-Store 아키텍처라고 부르며, RISC의 핵심 특징이다.

사용하는 명령어 형식 (I-type)
| opcode (6) | rs (5) | rt (5) | immediate (16-bit) |

여기서 rt는 데이터를 읽거나 쓸 레지스터, rs는 메모리 주소 계산에 사용하는 기준(base) 레지스터, immediate는 오프셋(offset)이다.

* 슬라이드 노트: rt = s1(destination/source1), rs = s2(base address)

Page 15

카테고리 2: 데이터 이동 명령어 (2) — lw와 sw 예제

lw(Load Word)는 메모리에서 레지스터로, sw(Store Word)는 레지스터에서 메모리로 32비트 데이터를 옮긴다.
예제 1: lw rt, 0xEDAC(rs) — Load Word
| opcode(100011) | rs | rt | immediate(0xEDAC) |

동작: GPR[rt] ← Memory[GPR[rs] + 0xEDAC]

기준 레지스터(rs)의 값에 오프셋 0xEDAC를 더한 메모리 주소에서 32비트(1 word)를 읽어 레지스터 rt에 저장한다.

예제 2: sw rt, 0xEDAC(rs) — Store Word
| opcode(101011) | rs | rt | immediate(0xEDAC) |

동작: Memory[GPR[rs] + 0xEDAC] ← GPR[rt]

lw와 반대 방향이다. 레지스터 rt의 값을 메모리 주소 (GPR[rs] + 0xEDAC)에 저장한다.

중요 사항
  • 워드 정렬(Word Alignment): 모든 계산된 주소는 워드(4바이트) 단위로 정렬되어야 한다. 즉, 주소는 4의 배수여야 한다.
  • 예외(Exception): 가상 주소에서 물리 주소로의 변환이 유효하지 않으면 예외가 발생한다.
  • 가상 주소(Virtual Address): 명령어가 보는 주소는 가상 주소이며, 실제 메모리 접근을 위해 물리 주소로 변환되어야 한다.
Page 16

카테고리 2: 데이터 이동 명령어 (3) — 비정렬 로드

일반적인 lw/sw는 워드 정렬된 주소만 접근할 수 있다. 정렬되지 않은 주소의 데이터를 읽으려면 LWL/LWR 명령어 쌍을 사용한다.
상세 설명

메모리에서 데이터를 읽을 때, 원하는 데이터가 워드 경계(4바이트 경계)에 걸쳐 있을 수 있다. 예를 들어, 주소 1~4에 걸친 4바이트를 읽고 싶을 때, 이는 두 워드(주소 0~3과 4~7)에 걸쳐 있다.

이런 경우를 위해 MIPS ISA는 특수 명령어 쌍을 제공한다:

  • LWL (Load Word Left): 비정렬 주소에서 워드의 상위(MSB) 부분을 로드
  • LWR (Load Word Right): 비정렬 주소에서 워드의 하위(LSB) 부분을 로드
예제: data[47:24] 즉 Byte 5~Byte 3을 로드하기 (Big Endian)

메모리 배치 (Big Endian):

주소 4: | Byte 7 | Byte 6 | Byte 5 | Byte 4 |
주소 0: | Byte 3 | Byte 2 | Byte 1 | Byte 0 |

Byte 5, 4, 3을 하나의 워드로 가져오려면:

Step 1) LWL rt, 5(rs)   // 상위 부분 로드 (Byte 5, 4를 rt 상위에 배치)
Step 2) LWR rt, 0(rs)   // 하위 부분 로드 (Byte 3을 rt 하위에 배치)

두 명령어를 순서대로 실행하면, rt에 원하는 비정렬 데이터가 완성된다.

비정렬 로드는 실무에서 네트워크 패킷 처리, 파일 파싱 등에서 자주 필요하다. 최신 MIPS ISA에서 제공하는 기능이며, 참고 문헌은 "MIPS IV Instruction Set, 1995"이다.

실습: 배열 접근 프로그램 컴파일

C 코드 A[12] = h + A[8]를 MIPS 어셈블리로 변환한다. 배열 원소 접근에 오프셋 계산이 핵심이다.
문제 설정

C 코드:

A[12] = h + A[8];

레지스터 배정:

  • r0: 정수 배열 A의 기준 주소(Base Address)
  • r1: 변수 h의 값
  • t0: 임시 레지스터
MIPS 어셈블리 코드
LW  t0, 32(r0)    // 32 = 8 * 4, A[8]을 메모리에서 로드
ADD t0, r1, t0     // t0 = h + A[8]
SW  t0, 48(r0)     // 48 = 12 * 4, 결과를 A[12]에 저장

오프셋 계산 방법:

정수(int)는 4바이트이므로, 배열의 n번째 원소의 오프셋은 n × 4이다.

  • A[8]의 오프셋: 8 × 4 = 32
  • A[12]의 오프셋: 12 × 4 = 48
배열 접근에서 가장 중요한 것은 바이트 주소 오프셋을 올바르게 계산하는 것이다. 원소 인덱스에 원소 크기(바이트 수)를 곱해야 실제 메모리 주소가 된다.

카테고리 3: 제어 흐름 명령어 (1) — 개요

제어 흐름(Control Flow) 명령어는 프로그램의 실행 순서를 바꾸는 명령어이다. R-type, I-type, J-type 세 가지 형식을 모두 사용한다.
상세 설명

지금까지의 ALU 명령어와 데이터 이동 명령어는 순차적으로 실행된다. 하지만 실제 프로그램에는 if-else, for, while, 함수 호출 등이 있어 실행 흐름을 바꿔야 한다.

제어 흐름 명령어가 사용하는 형식:

R-type: | 0 (고정) | rs | rt | rd | shamt | funct |         (JR 등)
I-type: | opcode   | rs | rt | immediate (16-bit) |  (BEQ, BNE 등)
J-type: | opcode   | immediate (26-bit)         |     (J, JAL)
Page 19

제어 흐름과 기본 블록 (Basic Block)

프로그램은 분기(branch)를 기준으로 기본 블록(Basic Block)들로 나눌 수 있다. 기본 블록은 분기 없이 순차적으로 실행되는 코드 구간이다.
상세 설명

프로그램에는 if-else 같은 분기(branch)가 있다. 분기는 프로그램의 "갈림길"이라고 생각할 수 있다.

기본 블록(Basic Block)이란, 분기가 없는 연속된 코드의 묶음이다. 한번 기본 블록에 진입하면, 블록 끝까지 순차적으로 실행된다.

예를 들어, if-else 구조에서:

  • 기본 블록 0: 조건 검사 전까지의 코드
  • 기본 블록 1: if가 참일 때 실행되는 코드
  • 기본 블록 2: else일 때 실행되는 코드

제어 흐름 명령어는 기본 블록 자체가 아니라 분기(갈림길)를 위해 정의된다.

핵심 용어
영어한국어설명
Branch분기프로그램 실행 흐름이 갈라지는 지점
Basic Block기본 블록분기 없이 순차 실행되는 코드 구간
Control Flow제어 흐름프로그램의 명령어 실행 순서

조건부 분기 (Conditional Branch) — BEQ

조건부 분기는 특정 조건이 참일 때만 분기하는 명령어이다. if-else, 루프 종료 조건 등에 사용된다.
예제: beq rs, rt, 0x0005
| opcode(BEQ) | rs | rt | Immediate(0x0005, signed) |

동작:

  1. GPR[rs] == GPR[rt] 인지 확인
  2. TRUE이면: BEQ 다음 명령어로부터 5개 명령어 떨어진 곳으로 분기
    • 새 PC = PC + 4 + (5 << 2) = PC + 4 + 20
  3. FALSE이면: 바로 다음 명령어 실행 (PC + 4)
왜 2비트 시프트(shift)를 할까?

MIPS의 모든 명령어는 4바이트(32비트) 크기이다. 따라서 명령어의 주소는 항상 4의 배수이다.

분기 오프셋에 4를 곱하면(= 왼쪽으로 2비트 시프트), 16비트 즉시값으로 실제로는 18비트 범위의 바이트 주소를 표현할 수 있다. 이렇게 하면 더 넓은 범위로 분기할 수 있다.

\[ \text{Target Address} = \text{PC} + 4 + (\text{offset} \times 4) \]
PC + 4인 이유: 분기 명령어의 다음 명령어(PC+4) 기준으로 오프셋을 계산한다. 이를 PC-relative addressing이라 한다.
Page 21

조건부 분기 확장 — BGT/BLE 없이 비교하기

MIPS는 BEQ, BNE, BLEZ, BGTZ를 제공하지만 BGT(>), BLE(≤)는 제공하지 않는다. 일반적인 대소 비교는 뺄셈 후 BLEZ/BGTZ로 구현한다.
제공되는 조건 분기 명령어
명령어조건설명
BEQrs == rtEqual: 같으면 분기
BNErs != rtNot Equal: 다르면 분기
BLEZrs ≤ 0Less than or Equal to Zero
BGTZrs > 0Greater Than Zero
a > b 또는 a < b를 구현하는 방법

핵심 아이디어: 뺄셈으로 변환한다.

a > b  →  (a - b) > 0  →  BGTZ (a-b) TARGET
a < b  →  (a - b) < 0  →  BLEZ (a-b) TARGET

먼저 SUB 명령어로 a - b를 계산한 후, 그 결과에 대해 BGTZ 또는 BLEZ를 사용한다.

설계 철학: BGT, BLE 등의 전용 명령어를 추가하지 않는 이유는 디코더의 복잡도를 줄이기 위해서이다. 뺄셈 한 번을 추가하는 것은 극히 작은 오버헤드(miniscule overhead)이므로, 명령어 수를 줄여 하드웨어를 단순하게 유지하는 것이 더 이득이다.

점프 명령어 (Jump) — J와 JR

점프(Jump)는 조건 없이 무조건 특정 주소로 이동하는 명령어이다. J는 즉시값 주소로, JR은 레지스터에 저장된 주소로 점프한다.
예제: j 25000 (J-type)
| opcode(J) | Immediate(25000, unsigned) |

동작: PC ← 25000 << 2 = 100000

26비트 즉시값에 4를 곱하여(2비트 좌시프트) 실제 바이트 주소를 만든다. 이로써 25000번째 명령어로 점프한다.

C 언어의 goto에 해당한다:

void main() {
    int i = rand();
    if (i == 5)
        goto report;
    return;

    report:
    printf("%d\n", i);
}
JR (Jump Register) — R-type

JR은 레지스터에 저장된 주소로 점프한다.

  • R-type 형식이지만 레지스터 1개만 사용 (rs 필드만 의미 있음)
  • 주로 JAL과 함께 사용 — 함수에서 호출자에게 돌아갈 때 JR $ra를 사용
핵심 용어
영어한국어설명
J (Jump)점프무조건 즉시값 주소로 이동 (J-type)
JR (Jump Register)레지스터 점프레지스터 값이 가리키는 주소로 이동 (R-type)
JALJump and LinkJ와 같지만 복귀 주소를 저장 (함수 호출)

실습: if-else 프로그램 컴파일

C 코드의 if-else 구조를 BNE, ADD, SUB, J 명령어를 사용하여 MIPS 어셈블리로 변환한다.
문제 설정

C 코드:

if (i == j)
    f = g + h;
else
    f = g - h;

레지스터 배정: f, g, h → r0, r1, r2 / 조건 변수 i, j → r3, r4

ELSE 분기의 레이블: L1

MIPS 어셈블리 코드
      BNE r3, r4, L1    // i != j이면 L1(ELSE)로 점프
      ADD r0, r1, r2    // f = g + h  (IF 케이스)
      J   Exit          // ELSE를 건너뛰고 종료로 점프
L1:   SUB r0, r1, r2    // f = g - h  (ELSE 케이스)
Exit:

실행 흐름:

  1. BNE로 i와 j를 비교 — 같지 않으면 L1(ELSE)로 분기
  2. 같으면(i==j): ADD 실행 후 J로 Exit로 점프하여 ELSE를 건너뜀
  3. 다르면(i!=j): L1의 SUB 실행
주의: "i == j"를 검사하는데 BEQ가 아닌 BNE를 사용한 이유에 주목하라. "같으면 IF를 실행"하려면, "같지 않으면 ELSE로 건너뛰기"로 뒤집어서 구현하는 것이 일반적이다. 조건을 반전(negate)시켜서 분기하는 패턴은 어셈블리에서 매우 흔하다.
Page 24

JAL — 점프와 함수 호출

JAL(Jump And Link)은 함수를 호출할 때 사용하는 명령어이다. 복귀 주소(PC+4)를 $ra(r31) 레지스터에 자동 저장하고, 지정된 주소로 점프한다.
함수 호출의 필요성

프로그램에서 함수 호출은 매우 흔하다. 함수를 호출하고 나면, 함수 실행이 끝난 후 원래 호출한 지점의 다음 명령어로 돌아와야 한다.

main:
  ...
  call foo    ← 함수 호출 (여기서 foo로 이동)
  ...         ← foo 실행 후 여기로 복귀해야 함

foo:
  ...         ← 실행 후 main으로 돌아가야 함
JAL 명령어 (J-type)
| opcode(JAL) | Immediate(25000, unsigned) |

동작:

  1. $ra(r31) ← PC + 4 (복귀 주소 저장)
  2. PC ← 25000 << 2 (목표 함수 주소로 점프)

J 명령어와 형식은 동일하지만, 점프 전에 복귀 주소를 $ra에 자동 저장하는 것이 핵심 차이이다.

함수에서 복귀하기

함수 실행이 끝나면 JR $ra로 호출자에게 돌아간다.

JAL foo       // $ra = PC+4, PC = foo의 주소
...
foo:
  ...         // 함수 본체
  JR $ra      // $ra에 저장된 주소(호출자의 다음 명령어)로 복귀
JAL/JR 조합은 함수 호출/복귀의 기본 메커니즘이다. 하지만 함수 안에서 또 다른 함수를 호출(중첩 호출)하면 $ra가 덮어써진다. 이 문제를 해결하기 위해 스택(Stack)을 사용한다 — 이후 페이지에서 다룬다.
Page 26

함수 호출 시 고려사항 (1)

함수 호출 시 호출자(caller)의 상태 보존, 인자 전달, 반환 값 전달이라는 세 가지 문제를 해결해야 한다.
상세 설명

다음 코드를 생각해 보자:

int main() {          // caller (호출자)
    int rv = 0;
    rv = foo(0);      // 함수 호출
    rv += 1;          // ← 복귀 지점 (Resume point)
    return rv;
}
int foo(int a) {      // callee (피호출자)
    return (a + 1);
}

여기서 세 가지 문제가 발생한다:

  1. 상태 보존: main의 변수 rv는 foo 실행 중에도 살아 있어야 한다. foo가 같은 레지스터를 사용하면 rv가 덮어써질 수 있다.
  2. 인자 전달: main이 foo에게 인자 0을 어떻게 전달할 것인가?
  3. 반환 값: foo가 계산한 결과(a+1)를 main이 어떻게 받을 것인가?
핵심 질문: "마이크로아키텍처가 함수 호출을 어떻게 지원할 것인가?" — 답은 레지스터에 역할을 부여하고, 스택 메모리를 활용하는 것이다.
Page 27

MIPS 레지스터 역할 지정

MIPS는 32개의 레지스터에 각각 특정 역할(Convention)을 부여하여 함수 호출 문제를 해결한다. 레지스터는 Caller-saved와 Callee-saved로 나뉜다.
주요 레지스터 역할
레지스터인덱스용도
$zero0항상 값이 0인 상수 레지스터
$v0 - $v12-3함수 반환 값 (Return values)
$a0 - $a34-7함수 인자 (Arguments)
$t0 - $t98-15, 24-25임시 레지스터 (Temporary) — Caller-saved
$s0 - $s716-23보존 레지스터 (Saved) — Callee-saved
$sp29스택 포인터 (Stack Pointer)
$fp30프레임 포인터 (Frame Pointer)
$ra31복귀 주소 (Return Address)
Caller-saved vs. Callee-saved

Caller-saved 레지스터 ($t0~$t9 등):

  • 호출자(caller)가 함수 호출 전에 이 레지스터를 저장해야 한다
  • 피호출자(callee)는 이 레지스터를 자유롭게 사용 가능

Callee-saved 레지스터 ($s0~$s7 등):

  • 피호출자(callee)가 이 레지스터를 사용하려면, 먼저 기존 값을 저장하고 사용 후 복원해야 한다
  • 호출자는 이 레지스터가 함수 호출 후에도 값이 보존됨을 보장받는다
비유: Caller-saved는 "내가 빌려주는 물건은 내가 챙겨야 한다", Callee-saved는 "빌린 사람이 원래 상태로 돌려놓아야 한다"와 같다.
Page 28

스택과 함수 프레임 (1) — 호출 전

메모리의 스택(Stack)은 함수 호출 시 아키텍처 상태를 보존하는 데 사용된다. $fp는 함수 프레임의 경계를, $sp는 스택의 현재 끝을 가리킨다.
예제 코드와 레지스터 배정
int main() {
    int rv = 0;
    ...
    rv = foo(89);
    rv += 1;
    return rv;
}
int foo(int a) {
    int j = 7;
    return (a + j);
}

레지스터 배정:

  • main::rv → $s0
  • foo::a → $a0 (인자 레지스터)
  • foo::j → $s0 (saved 레지스터 — main의 rv와 충돌!)
호출 전 스택 상태
높은 주소 (High Address)
┌──────────────────┐
│  Stack of main   │ ← $fp (프레임 포인터)
│                  │ ← $sp (스택 포인터)
└──────────────────┘
낮은 주소 (Low Address)

foo가 호출되기 전, 스택에는 main의 프레임만 있다. $fp와 $sp가 main 프레임의 경계를 가리킨다.

문제: foo도 $s0을 사용하려 하지만, main의 rv 값이 $s0에 있다. foo가 $s0을 사용하면 rv 값이 사라진다! 이 문제를 스택을 사용하여 해결한다 — 다음 페이지에서 확인.
Page 29

스택과 함수 프레임 (2) — 호출 후

foo가 호출되면 스택에 새로운 프레임이 생긴다. 기존 레지스터 값들($a0, $ra, $fp, $s0)을 스택에 저장한 후, foo의 본체를 실행한다.
호출 후 스택 상태
높은 주소 (High Address)
┌──────────────────┐
│  Stack of main   │ ← 이전 $fp
├──────────────────┤
│  Old value of $a0│  ← $sp + 12
│  Return address  │  ← $sp + 8   ($ra 저장)
│  Old value of $fp│  ← $sp + 4
│  Old value of $s0│  ← $sp + 0   (main의 rv 보존!)
└──────────────────┘
                     ← $sp (새로운 스택 끝)
낮은 주소 (Low Address)
foo의 핵심 실행 코드

스택에 기존 값을 저장한 뒤, foo의 실제 연산을 수행한다:

ADDI $a0, $zero, 89    // 인자 설정: a = 89
ADDI $s0, $zero, 7     // j = 7
ADD  $v0, $a0, $s0     // 반환값 = a + j = 96

$v0에 결과를 넣으면, main은 함수 복귀 후 $v0에서 반환 값을 가져올 수 있다.

스택에 저장하는 4가지 정보: (1) $a0 — 기존 인자 보존, (2) $ra — 복귀 주소 보존, (3) $fp — 이전 프레임 포인터 보존, (4) $s0 — callee-saved 레지스터 보존. 함수 종료 시 이들을 역순으로 복원한다.
Page 30

실습: 함수 호출 전체 컴파일

main에서 foo를 호출하는 전체 과정을 MIPS 어셈블리로 컴파일한다. 프롤로그(prologue)에서 레지스터를 저장하고, 에필로그(epilogue)에서 복원하는 패턴을 확인한다.
C 원본 코드
int main() {
    int rv = 0;
    rv = foo(89);
    rv += 1;
    return rv;
}
int foo(int a) {
    int j = 7;
    return (a + j);
}
전체 MIPS 어셈블리
main:
    ADDI $s0, $zero, 0       // rv = 0
    JAL  foo                  // foo 호출 ($ra에 복귀 주소 자동 저장)
    ADDI $s0, $v0, 1         // rv = foo의 반환값 + 1
    ADDI $v0, $zero, 0       // main의 반환값 = 0

foo:
    // ===== 프롤로그 (Prologue): 레지스터 저장 =====
    ADDI $sp, $sp, -16       // 스택에 16바이트(4워드) 공간 확보
    SW   $a0, 12($sp)        // 기존 $a0 저장
    SW   $ra, 8($sp)         // 복귀 주소 저장
    SW   $fp, 4($sp)         // 기존 프레임 포인터 저장
    SW   $s0, 0($sp)         // 기존 $s0 저장 (main의 rv 보존!)
    ADDI $fp, $sp, 16        // $fp 업데이트 (프레임 경계 설정)

    // ===== 함수 본체 =====
    ADDI $a0, $zero, 89      // 인자 a = 89
    ADDI $s0, $zero, 7       // j = 7
    ADD  $v0, $a0, $s0       // 반환값 = a + j = 96

    // ===== 에필로그 (Epilogue): 레지스터 복원 =====
    LW   $a0, 12($sp)        // $a0 복원
    LW   $ra, 8($sp)         // 복귀 주소 복원
    LW   $fp, 4($sp)         // 프레임 포인터 복원
    LW   $s0, 0($sp)         // $s0 복원 (main의 rv 되돌림)
    ADDI $sp, $sp, 16        // 스택 공간 반환

    JR   $ra                  // main으로 복귀
실행 흐름 분석
  1. main: $s0에 0을 저장 (rv = 0)
  2. JAL foo: $ra에 "ADDI $s0, $v0, 1"의 주소를 저장하고 foo로 점프
  3. foo 프롤로그: $sp를 16 감소시키고 4개 레지스터를 스택에 저장
  4. foo 본체: 인자 설정, 연산, 결과를 $v0에 저장
  5. foo 에필로그: 4개 레지스터를 스택에서 복원, $sp를 원복
  6. JR $ra: main의 JAL 다음 명령어로 복귀
  7. main 계속: $v0(96)에 1을 더해 $s0(rv)에 저장 → rv = 97
"우리는 컴파일러에게 감사해야 한다!(We need to thank compiler!!!)" — 이 복잡한 프롤로그/에필로그 코드를 프로그래머가 매번 작성할 필요 없이, 컴파일러가 자동으로 생성해준다.

RISC-V (UC Berkeley, 2014)

RISC-V는 UC Berkeley에서 2014년에 만든 오픈소스 ISA이다. ARM/MIPS와 달리 무료로 사용할 수 있어 학계와 산업계에서 빠르게 확산되고 있다.
상세 설명
  • 라이선스 문제 해결: ARM이나 MIPS ISA를 사용하려면 마이크로아키텍처 개발을 위해 로열티(loyalty)를 지불해야 한다. RISC-V는 오픈소스이므로 무료로 사용 가능하다.
  • MIPS와의 차이: MIPS는 opcode와 function 필드가 명령어의 MSB(상위 비트)에 위치하지만, RISC-V는 LSB(하위 비트)에 배치한다.
  • 오픈소스 마이크로아키텍처: ISA뿐 아니라 실제 하드웨어 구현도 오픈소스로 제공된다.
    • Rocket Chip: UC Berkeley에서 개발한 오픈소스 RISC-V 프로세서
    • CVA6: 또 다른 오픈소스 RISC-V 프로세서 구현
RISC-V는 학계에서 컴퓨터 구조 연구의 표준 플랫폼으로 자리잡아 가고 있으며, 산업계에서도 IoT, 임베디드 등에 활발히 채택되고 있다.

x86 ISA 개요

x86은 대표적인 CISC ISA로, 명령어 디코딩이 매우 복잡하다. 필드 길이가 opcode에 따라 가변적이며, 일부 필드는 아예 생략될 수 있다.
상세 설명

MIPS(RISC)와 대비되는 x86(CISC)의 특징:

  • 복잡한 명령어 디코딩: 명령어가 가변 길이(1바이트~15바이트)이므로 디코더 설계가 매우 어렵다
  • 가변 길이 필드: opcode에 따라 일부 필드의 길이가 달라진다
  • 0바이트 필드: opcode에 따라 특정 필드가 완전히 무시(생략)될 수 있다

x86 명령어의 일반적인 구조는 다음과 같이 여러 선택적 필드로 구성된다:

  • Instruction Prefix(선택)
  • Opcode (1~3 바이트)
  • ModR/M (선택)
  • SIB (선택)
  • Displacement (선택, 0/1/2/4 바이트)
  • Immediate (선택, 0/1/2/4 바이트)

참고: Intel, "Intel 64 and IA-32 Architectures Software Developer's Manual Volume 1: Basic Architecture," 2016.

x86 ISA: 레지스터

x86은 범용 레지스터(General-Purpose Registers), 세그먼트 레지스터, EFLAGS, 명령어 포인터 등 다양한 레지스터를 가진다. 프로그래머에게 보이지 않는 레지스터도 많다.
x86 레지스터 분류
레지스터 종류용도
General-Purpose RegistersALU 연산의 오퍼랜드, 주소 계산에 사용. ESP(스택 포인터), EBP(프레임 포인터) 등 MIPS의 $sp, $fp에 대응하는 레지스터 포함.
Segment Registers현재 프로그램이 사용하는 다양한 메모리 세그먼트를 가리키는 포인터
EFLAGS프로그램 상태 및 제어 레지스터 — 오버플로우, 제로 플래그 등 조건 정보를 담음
Instruction Pointer (EIP)MIPS의 PC와 동일 — 다음에 실행할 명령어의 주소를 가리킴
위의 레지스터들은 프로그래머에게 보이는 "마이크로아키텍처 상태"일 뿐이다. 실제 프로세서 내부에는 프로그램이 볼 수 없는 훨씬 많은 레지스터가 존재한다 (예: 리네이밍 레지스터, 내부 버퍼 등).

x86 디코딩 (1) — ADD 명령어의 opcode

x86의 ADD 명령어를 예로 들어, opcode 비트 하나하나가 동작 방향(d)과 오퍼랜드 크기(s)를 결정하는 과정을 살펴본다.
상세 설명

x86 ADD의 opcode: 0000_00ds

ADD의 기본 동작: destination = destination + source

  • d 비트 (Direction):
    • d=0: 목적지가 메모리, 소스가 레지스터
    • d=1: 목적지가 레지스터, 소스가 메모리
  • s 비트 (Size):
    • s=0: 8비트 오퍼랜드끼리 덧셈
    • s=1: 32비트 오퍼랜드끼리 덧셈
MIPS와의 비교

MIPS에서는 ADD가 항상 레지스터 → 레지스터 연산이었다. 메모리 접근은 별도의 LW/SW 명령어가 담당했다 (Load-Store 아키텍처).

x86에서는 ADD 하나의 명령어 안에서 메모리 접근과 연산을 동시에 수행할 수 있다. 이것이 CISC의 특징이다 — 명령어 하나가 더 많은 일을 하지만, 디코딩이 복잡해진다.

Page 36

x86 디코딩 (2) — ModR/M 바이트

x86 명령어에서 오퍼랜드의 위치를 지정하는 ModR/M 바이트는 MOD(2비트), REG(3비트), R/M(3비트) 필드로 구성되며, 조합에 따라 매우 다양한 주소 지정 모드를 지원한다.
ModR/M 바이트의 구조
필드비트역할
MOD2비트주소 지정 모드 선택
REG3비트보통 레지스터 인덱스 (eax, ebx 등)
R/M3비트레지스터 또는 메모리 주소 지정 방식
MOD 필드의 의미
  • MOD = 00: displacement 비트를 보통 무시
    • 예외: R/M = 101이면 32비트 displacement를 메모리 주소로 사용
  • MOD = 01: 8비트 displacement 사용
  • MOD = 10: 32비트 displacement 사용
  • MOD = 11: R/M 필드가 레지스터를 인덱싱 (메모리 주소 지정이 아님)
R/M 필드의 의미
  • 0xx, 101, 110, 111 (MOD != 11일 때): 해당 레지스터 값을 사용하여 메모리 주소를 계산
  • 100: SIB(Scale-Index-Base) 필드를 사용하여 메모리 주소를 계산 — 더 복잡한 주소 지정 가능
MIPS와의 차이: MIPS는 고정된 3가지 형식(R/I/J)만 있으면 되지만, x86은 MOD/REG/R/M 조합만으로도 수십 가지 경우의 수가 발생한다. 여기에 SIB, Prefix까지 고려하면 디코더 설계가 기하급수적으로 복잡해진다.
Page 37-38

x86 디코딩 (3)-(4) — 디코딩 테이블

x86의 ModR/M 바이트의 모든 조합을 나타내는 디코딩 참조 테이블이다. MOD, REG, R/M의 조합에 따라 실제 오퍼랜드가 결정되는 방식을 보여준다.
상세 설명

슬라이드 37~38페이지는 x86 명령어의 ModR/M 디코딩 테이블을 보여준다. 이 테이블은 MOD(행)와 R/M(열) 값의 조합에 따라 어떤 레지스터나 메모리 주소 지정 방식이 사용되는지를 일목요연하게 정리한 것이다.

예를 들어:

  • MOD=00, R/M=000 → [EAX] — EAX 레지스터가 가리키는 메모리 주소
  • MOD=00, R/M=101 → 32비트 즉시 주소 (disp32)
  • MOD=01, R/M=000 → [EAX + disp8] — EAX + 8비트 오프셋
  • MOD=11, R/M=000 → EAX/AL — 레지스터 자체를 직접 사용
핵심 용어
영어한국어설명
ModR/M모드/레지스터/메모리오퍼랜드 주소 지정을 결정하는 1바이트 필드
SIBScale-Index-Base복잡한 주소 계산을 위한 추가 필드 (base + index*scale + disp)
Displacement변위메모리 주소 계산에 더해지는 오프셋 값
핵심 메시지: x86(CISC)의 디코딩이 MIPS(RISC)에 비해 얼마나 복잡한지를 이 테이블이 극명하게 보여준다. MIPS는 opcode 6비트만 보면 명령어 형식을 바로 알 수 있지만, x86은 opcode, ModR/M, SIB를 순차적으로 해석해야 비로소 오퍼랜드를 알 수 있다. 이것이 RISC의 디코딩 단순성이 가지는 하드웨어 설계 이점이다.
