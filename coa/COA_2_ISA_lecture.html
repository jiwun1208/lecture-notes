Instruction Set Architecture: Introduction

이 강의는 명령어 집합 구조(ISA, Instruction Set Architecture)의 개념과 기본 구성 요소를 소개하는 챕터이다. 컴퓨터가 프로그램을 어떻게 이해하고 실행하는지의 핵심 인터페이스를 다룬다.
강의 개요

ISA란 소프트웨어와 하드웨어 사이의 약속(인터페이스)이다. 프로그래머가 작성한 코드가 컴퓨터 하드웨어에서 실행되려면, 양측이 합의한 명령어 형식이 필요하다. 이 강의에서는 ISA가 왜 필요한지, 프로그램이 어떻게 실행되는지, ISA에 어떤 정보가 담기는지, 그리고 ISA가 어떻게 발전해 왔는지를 순서대로 배운다.

아키텍처 vs 조직: 복습 (Architecture vs. Organization)

아키텍처(Architecture)는 "무엇을 할 수 있는가"를 정의하고, 마이크로아키텍처(Organization/Microarchitecture)는 "그것을 어떻게 구현하는가"를 다룬다.
상세 설명

아키텍처 수준(Architecture level)은 회로 설계의 전문가가 아니어도 이해할 수 있는 추상화 단계이다. 자동차에 비유하면, 운전자 매뉴얼에 해당한다. 자동차가 "전진, 후진, 좌회전, 우회전"을 할 수 있다는 것은 알지만, 엔진 내부 구조까지 알 필요는 없다.

마이크로아키텍처 수준(Organization level)은 특정 데이터 경로(data path)와 제어 장치(control unit)를 설계하는 단계이다. 자동차로 비유하면, 특정 엔진과 프레임을 선택하여 실제로 차가 달리게 만드는 것이다.

아키텍처가 수행할 수 있는 기능의 예시:

  • 산술 연산 (Arithmetic operations)
  • 논리 연산 (Logical operations)
  • 데이터 적재 및 저장 (Load & Store)
  • 입출력 장치와의 상호작용 (Interact with I/O)

이러한 기능들을 형식화된 방식으로 정의하면, 그것이 바로 ISA가 된다. 핵심 질문: "아키텍처를 좀 더 공식적(formal)인 방법으로 정의하려면 어떻게 해야 할까?"

핵심 용어

시험 포인트: Architecture와 Organization의 차이를 자동차 비유로 설명할 수 있어야 한다. Architecture = 운전 매뉴얼(what), Organization = 엔진 설계(how).

명령어 집합 구조란? (Instruction Set Architecture)

ISA는 시스템 소프트웨어와 하드웨어 조직 사이의 인터페이스이며, 형식화된 명령어의 집합으로 아키텍처를 기술한다.
상세 설명

아키텍처의 특성을 생각해 보자:

  • 아키텍처는 프로그래머와 하드웨어 설계자 양쪽이 바라보는 시스템의 모습이다 (Amdahl et al., "Architecture of the IBM System/360," 1964).
  • 아키텍처는 시스템 소프트웨어와 하드웨어 조직 사이의 인터페이스(interface) 역할을 한다.

명령어(instruction)는 시스템의 주요 기능을 간결하게 표현할 수 있다:

  • 하드웨어 설계자 입장: ISA를 보고 RTL(Register Transfer Level)에서 무엇을 구현해야 하는지 결정한다.
  • 컴파일러 & 프로그래머 입장: ISA를 보고 소프트웨어를 극한까지 최적화할 수 있다.

결론적으로, 아키텍처를 기술하기 위해 형식화된 명령어의 집합, 즉 ISA를 정의해야 한다. ISA는 소프트웨어 엔지니어와 하드웨어 엔지니어가 공통으로 참조하는 계약서와 같다.

ISA = 소프트웨어와 하드웨어 사이의 계약서(contract). 하드웨어 설계자는 이 계약에 맞춰 회로를 만들고, 컴파일러/프로그래머는 이 계약에 맞춰 코드를 생성한다.
핵심 용어
영어한국어설명
ISA명령어 집합 구조아키텍처를 정의하는 형식화된 명령어들의 집합
RTL레지스터 전송 수준하드웨어 설계의 추상화 수준, 레지스터 간 데이터 전달을 기술
Interface인터페이스두 계층 사이의 상호작용 규약

프로그래밍 복습 (Remind of Programming)

C 프로그램의 개발 과정은 작성 → 컴파일 → 링크 → 실행의 4단계를 거치며, 이 과정에서 ISA가 핵심 역할을 한다.
상세 설명

C 프로그램이 실행되기까지의 절차를 하나씩 살펴보자:

  1. C 프로그램 작성 (*.c 파일): 사람이 읽을 수 있는 고수준 언어로 코드를 작성한다.
  2. 컴파일러(Compiler)가 C 코드를 어셈블리 프로그램 (*.s 파일)으로 변환한다. 이 어셈블리 코드가 바로 ISA로 기술된 것이다. GCC가 이 역할을 수행한다.
  3. 어셈블러(Assembler)가 어셈블리를 오브젝트 파일 (*.o)로 변환한다. 이 단계에서 기계가 인식할 수 있는 바이너리 형태가 된다.
  4. 링커(Linker, ld)가 오브젝트 파일과 라이브러리 오브젝트(*.o, *.so)를 합쳐 실행 파일 (*.exe, *.elf 등)을 만든다.
  5. 로더(Loader)가 실행 파일을 메모리에 적재하여 프로그램을 실행한다.

중요한 점: C 코드 자체는 기계가 직접 이해할 수 없고, 컴파일 과정을 거쳐 ISA 형태의 어셈블리로 변환되어야 기계가 인식할 수 있다.

핵심 용어
영어한국어설명
Compiler컴파일러고수준 언어를 어셈블리/기계어로 번역하는 프로그램
Assembler어셈블러어셈블리 코드를 기계어(오브젝트 파일)로 변환
Linker링커여러 오브젝트 파일을 하나의 실행 파일로 합침
Loader로더실행 파일을 메모리에 적재하는 OS 구성 요소
Object File오브젝트 파일기계어로 변환된 중간 바이너리 파일 (*.o)
시험 포인트: "ISA로 기술된(Described as ISA)" 단계가 어디인지 기억하라. 어셈블리 프로그램(*.s)이 바로 ISA의 표현이다.

예제: 다중 C 프로그램 (Multiple C Programs)

두 개의 C 소스 파일(main.c, calculate.c)로 구성된 프로그램 예제를 통해 전역 변수, 외부 함수, 정적 변수 등의 개념을 복습한다.
코드 예제

main.c: 전역 변수를 기반으로 외부 함수를 호출하여 계산을 수행한다.

int buf[2] = {4, 7};        // 전역 변수 (.data 섹션)
void main() {
    int result;              // 지역 변수
    result = calculate(buf); // 외부 함수 호출
    printf("%d\n", result);
}

calculate.c: 인자와 정적 변수를 기반으로 계산을 수행한다.

static int offset;           // 정적 변수 (.bss 섹션)
int calculate(int* buf) {    // 지역 변수
    return (buf[0] * buf[1] + offset);
}
핵심 개념

전역 변수(Global variable): buf[2]는 프로그램 전체에서 접근 가능하며, 초기값이 있으므로 .data 섹션에 저장된다.

정적 변수(Static variable): offset은 해당 파일(calculate.c) 내에서만 접근 가능하며, 초기값이 없으므로 .bss 섹션에 저장된다. .bss 섹션은 "초기화되지 않은 데이터" 영역이다.

지역 변수(Local variable): result는 main 함수 내에서만 유효하며, 스택(stack)에 저장된다.

외부 함수(External function): main.c에서 calculate()를 호출하지만, 이 함수는 calculate.c에 정의되어 있다. 컴파일 시점에는 이 함수의 위치를 모르므로, 링킹 단계에서 해결해야 한다.

핵심 용어
영어한국어설명
.data데이터 섹션초기화된 전역/정적 변수가 저장되는 영역
.bssBSS 섹션초기화되지 않은 전역/정적 변수가 저장되는 영역
.text텍스트 섹션실행 가능한 기계어 코드가 저장되는 영역
static variable정적 변수해당 파일 내에서만 접근 가능한 변수

다중 파일 컴파일 (Compiling Multiple Program Files)

GCC 컴파일러를 사용하면 여러 C 파일을 한 번에 컴파일하고 링킹하여 하나의 실행 파일을 만들 수 있다.
상세 설명

GCC(GNU Compiler Collection)는 컴파일, 어셈블, 링킹을 모두 수행하는 종합 컴파일러이다.

사용 형식:

gcc -o <대상 실행 파일> <입력 파일들>

예시:

hklee@server> gcc -o calc main.c calculate.c
hklee@server> ./calc
28

실행 과정을 도식으로 표현하면:

  1. main.c와 calculate.c가 각각 컴파일러 & 어셈블러를 거쳐 main.o와 calculate.o로 변환된다.
  2. 링커(Linker)가 두 오브젝트 파일을 합쳐 실행 파일 calc를 생성한다.
  3. Linux에서는 실행 파일의 확장자가 크게 중요하지 않다 (Windows의 .exe와 달리).

결과값 28이 나오는 이유: buf[0] * buf[1] + offset = 4 * 7 + 0 = 28 (offset은 .bss에 있으므로 기본값 0).

링킹이 왜 필요한가? (Why Do We Need Linking?)

개별 오브젝트 파일은 다른 파일의 데이터와 코드 위치를 알 수 없으므로, 링킹(Linking)을 통해 코드와 데이터를 하나의 파일로 재배치해야 한다.
상세 설명

저장 프로그램 컴퓨터(Stored-program computer)에서는 프로그램 코드가 메모리에 적재되어 실행된다. 코드와 데이터는 선형 메모리 주소(linear memory address)를 통해 접근한다.

문제가 되는 상황:

  • main.o는 calculate() 함수가 어디에 있는지 모른다. ("Where is calculate()")
  • 각 오브젝트 파일은 다른 파일의 피연산자(operand) 위치를 모른다. ("Where are operands?")

각 오브젝트 파일은 독립적으로 컴파일되었기 때문에, 서로의 존재를 인식하지 못한다. main.o에는 .data 섹션(buf)과 .text 섹션(main 함수)이 있고, calculate.o에는 .bss 섹션(offset)과 .text 섹션(calculate 함수)이 있다.

링킹의 핵심 목적: 독립적인 오브젝트 파일들의 코드와 데이터를 하나의 오브젝트 파일로 재배치(relocate)하여, 모든 참조가 올바른 주소를 가리키게 만드는 것이다.

코드와 데이터의 재배치 (Relocation of Codes and Data)

링커는 개별 오브젝트 파일의 코드와 데이터를 하나의 실행 파일(ELF)로 합치고, 시스템 코드도 함께 링킹한다. 로더가 이 실행 파일을 메모리에 적재하여 실행한다.
상세 설명

링커(Linker)가 수행하는 작업:

  1. main.o와 calculate.o의 .text, .data, .bss 섹션을 합친다.
  2. 시스템 코드(static system codes)와 시스템 데이터도 함께 링킹한다. 예를 들어, printf()나 메모리 할당 관련 시스템 코드가 포함된다.
  3. 최종적으로 하나의 실행 가능 오브젝트 파일(Executable object file)이 생성된다.

실행 파일의 구조:

  • ELF 헤더(ELF Header): 파일의 메타 정보
  • .text 섹션: 시스템 코드 + main() + calculate() + 기타 시스템 코드
  • .data 섹션: 시스템 데이터 + int buf[2] = {4, 7}
  • .bss 섹션: int offset = 87 (슬라이드에서 예시 값)

이 실행 파일을 로더(Loader)가 메모리에 적재하면 비로소 프로그램이 실행된다.

오브젝트 파일의 메모리 로딩 (Loading Object File to Memory)

OS의 로더가 실행 파일을 메모리에 적재하면, 프로그램은 코드/데이터 영역 외에 스택, 힙 등 동적 관리 영역도 할당받는다.
상세 설명

로더(Loader)는 OS의 일부로, 실행 파일의 각 섹션을 메모리의 적절한 위치에 배치한다. 프로그램 실행 시 메모리 레이아웃:

  • .text 섹션 (낮은 주소): 실행 가능한 코드
  • .data & .bss 섹션: 초기화된/초기화되지 않은 전역/정적 변수
  • 힙(Heap) (위로 성장): malloc() 등으로 동적 할당되는 메모리 영역
  • 공유 라이브러리 매핑 영역: 동적 링킹된 라이브러리가 매핑되는 영역
  • 사용자 스택(User Stack) (아래로 성장): 함수 호출 시 지역 변수, 복귀 주소 등이 저장
  • 커널 공간(Kernel space) (높은 주소, 0xC0000000~): OS가 사용하는 영역

핵심 포인트: 힙은 위로 성장(grows up)하고, 스택은 아래로 성장(grows down)한다. 각 영역은 실행 중에 동적으로 크기가 변한다.

핵심 용어
영어한국어설명
Heap힙동적 메모리 할당 영역, 위쪽으로 성장
Stack스택함수 호출/지역 변수 관리 영역, 아래쪽으로 성장
Kernel Space커널 공간OS 전용 메모리 영역, 사용자 프로그램은 직접 접근 불가
Shared Library공유 라이브러리여러 프로그램이 공유하는 라이브러리 (*.so 파일)
주의: 메모리 주소 0x0 근처는 일반적으로 NULL 포인터 접근 방지를 위해 비워 둔다. 실제 .text 섹션은 0x0보다 약간 위에서 시작한다.

ISA가 명시해야 할 정보 (Information Must be Specified by ISA)

ISA는 최소한 명령어(Instructions), 프로그래머 가시 상태(Programmer Visible States), 데이터 형식 및 크기(Data Format & Size)를 정의해야 한다.
상세 설명

시스템 상태를 고려하면서 데이터를 처리해야 하므로, ISA에는 최소한 다음 세 가지 정보가 필요하다:

  1. 명령어(Instructions)
    • 어떤 종류의 기능을 수행할 수 있는가?
    • 이 명령어들을 어떻게 분류할 것인가?
  2. 프로그래머 가시 상태(Programmer Visible States)
    • 명령어가 소비(consume)할 수 있는 가시적 정보란 무엇인가?
    • 프로그램이 볼 수 있는 정보는 무엇인가? (레지스터, 메모리, PC 등)
  3. 데이터 형식 및 크기(Data Format & Size)
    • 데이터 타입은 무엇인가?
    • 문자(Char)? 부동소수점(Floating point)? 정수(Integer)? 부호 있는(Signed)?
ISA를 설계한다는 것은 "어떤 연산을 지원하고, 어떤 상태를 볼 수 있으며, 어떤 데이터를 다루는가"를 공식적으로 정의하는 것이다.

명령어의 일반적 분류 (General Classes of Instructions)

ISA의 명령어는 크게 산술/논리 명령어, 데이터 이동 명령어, 제어 흐름 명령어의 세 가지로 분류된다.
상세 설명

일반적으로 ISA의 명령어는 세 가지 주요 클래스로 나뉜다:

  1. 산술 및 논리 명령어(Arithmetic and Logical Instructions): 덧셈, 뺄셈, AND, OR 등의 연산 수행
  2. 데이터 이동 명령어(Data Movement Instructions): 데이터를 한 곳에서 다른 곳으로 이동 (레지스터 ↔ 메모리)
  3. 제어 흐름 명령어(Control Flow Instructions): 프로그램 실행 순서를 변경 (분기, 점프 등)

중요한 속성으로 원자성(Atomicity)이 있다:

  • 각 명령어의 실행은 다른 명령어에 의해 방해받을 수 없다.
  • 명령어는 다른 명령어의 최종 결과만 소비할 수 있고, 중간 값(intermediate value)은 소비할 수 없다.
핵심 용어
영어한국어설명
Atomicity원자성명령어 실행이 중간에 끊기지 않고 완전히 수행되는 속성

산술 및 논리 명령어 (Arithmetic and Logical Instructions)

산술/논리 명령어는 피연산자를 가져와서 연산을 수행하고, 결과를 프로그래머 가시 상태에 저장하는 세 단계를 거친다.
상세 설명

산술/논리 명령어의 대표적인 예: Add(덧셈), Subtract(뺄셈), AND, OR 등.

기본 흐름(3단계):

  1. 피연산자 로드(Load operands): 프로그래머 가시 상태(레지스터, 메모리 등)에서 피연산자를 가져온다.
  2. 연산 수행(Compute): 가산기(Adder) 등의 하드웨어를 사용하여 피연산자에 대한 연산을 수행한다.
  3. 결과 저장(Store result): 결과를 프로그래머 가시 상태의 특정 위치에 저장하여, 다음 명령어가 소비할 수 있게 한다.

이 과정을 간단히 비유하면: 서랍(레지스터)에서 숫자 두 개를 꺼내고, 계산기(ALU)로 더한 다음, 결과를 다시 서랍에 넣는 것이다.

Page 17

데이터 이동 명령어 (Data Movement Instructions)

데이터 이동 명령어는 한 가시 상태에서 다른 가시 상태로 데이터를 옮기는 역할을 한다.
상세 설명

대표적인 예: Move(이동), Load(적재), Store(저장).

기본 흐름:

  1. 피연산자 가져오기(Fetch operands): 프로그래머 가시 상태(소스 레지스터 또는 메모리)에서 데이터를 가져온다.
  2. 데이터 저장(Store operand): 특정 가시 상태(목적지 레지스터 또는 메모리)에 데이터를 저장한다.

데이터 흐름의 방향:

  • Register → Register: 레지스터 간 이동 (move)
  • Memory → Register: 메모리에서 레지스터로 적재 (load)
  • Register → Memory: 레지스터에서 메모리로 저장 (store)

산술 명령어와 달리 "연산"이 없고, 단순히 데이터를 한 곳에서 다른 곳으로 옮기는 것이 핵심이다.

제어 흐름 명령어 (Control Flow Instructions)

제어 흐름 명령어는 프로그램의 실행 순서를 변경하며, for/while 루프나 if-else 분기를 구현하는 데 사용된다.
상세 설명

대표적인 예: Branch(분기), Jump(점프).

프로그램은 기본적으로 위에서 아래로 순차 실행되지만, 반복이나 조건 분기가 필요할 때 제어 흐름 명령어가 사용된다. C 언어의 for, while 루프와 if-else 문이 이 명령어들로 구현된다.

기본 흐름:

  1. 피연산자 가져오기(Fetch operands): 프로그래머 가시 상태에서 비교할 값을 가져온다.
  2. 분기 조건 및 목표 주소 계산(Compute branch condition and target address): 조건이 참인지 판단하고, 점프할 코드의 주소를 계산한다.
  3. 조건부 점프(Conditional jump): 조건이 참이면 목표 코드로 점프하고, 거짓이면 다음 명령어를 계속 실행한다.

그림으로 표현하면: 조건(Condition)을 확인한 후, 참이면 "Codes of true" 경로로, 거짓이면 "Codes of false" 경로로 분기한다.

명령어의 피연산자 (Operands in Instructions)

명령어를 설계할 때 피연산자의 개수, 위치, 변형이라는 세 가지 관점에서 결정해야 한다. 이 선택이 CISC와 RISC의 차이를 만든다.
상세 설명

1. 피연산자 개수 (How many operands?)

  • 단항(Monadic) — 피연산자 1개: OPERATION <operand> (예: EDSAC)
  • 이항(Binatic) — 피연산자 2개: OPERATION <operand1> <operand2> (예: IBM 360)
  • 삼항(Triadic) — 피연산자 3개: OPERATION <operand1> <operand2> <operand3> (예: MIPS/ARM)

2. 피연산자 위치 (Where are operands?)

  • 메모리 또는 레지스터에 있을 수 있음 (x86/CISC) → 코드가 더 간결(compact)해짐
  • 레지스터에만 있을 수 있음 (RISC) → 메모리 데이터는 먼저 load해야 하므로 코드가 길어짐

3. 피연산자 변형 (How many variations?)

  • 다양함(Various) — CISC: opcode에 따라 피연산자의 개수와 길이가 달라짐
  • 적음(Few) — RISC: 피연산자의 개수와 길이가 거의 고정
핵심 용어
영어한국어설명
Monadic단항피연산자가 1개인 명령어 형식
Binatic이항피연산자가 2개인 명령어 형식
Triadic삼항피연산자가 3개인 명령어 형식
Opcode연산 코드명령어에서 수행할 연산의 종류를 지정하는 부분
시험 포인트: CISC는 피연산자가 메모리에도 있을 수 있어 코드가 짧지만 디코딩이 복잡하고, RISC는 레지스터만 사용하여 코드가 길지만 디코딩이 단순하다.

프로그래머 가시 상태 (Programmer Visible States)

프로그래머 가시 상태란 프로그램이 외부 세계 및 컴퓨터와 상호작용할 수 있는 메모리, 레지스터, 프로그램 카운터(PC) 등을 말한다.
상세 설명

아키텍처 상태(Architectural States)라고도 불리는 프로그래머 가시 상태는 프로그램이 "볼 수 있는" 모든 하드웨어 자원이다:

  • 메모리(Memory): M[0]부터 M[N-1]까지의 선형 주소 공간. 코드와 데이터를 저장한다.
  • 레지스터 파일(Register File, RF): reg0, reg1, ..., reg(R-1)까지의 빠른 저장소. 각 레지스터는 고정 비트 폭(예: 64비트)을 가진다.
  • 프로그램 카운터(Program Counter, PC): 현재 실행 중인 명령어의 주소를 저장하는 특수 레지스터. 예: PC가 0x4A8이면 메모리 주소 0x4A8에 있는 명령어를 실행 중이다.

명령어는 이 가시 상태들을 어떻게 조작하는지를 정의한다. ALU(산술논리연산장치)가 레지스터의 데이터를 처리하고, 결과를 다시 레지스터나 메모리에 저장한다. PC는 명령어를 실행하면서 자동으로 다음 명령어 주소로 갱신된다.

핵심 용어
영어한국어설명
Architectural State아키텍처 상태프로그래머가 볼 수 있는 하드웨어 상태의 총칭
Register File (RF)레지스터 파일CPU 내부의 고속 저장소 집합
Program Counter (PC)프로그램 카운터현재 실행할 명령어의 메모리 주소를 저장하는 레지스터
ALU산술논리연산장치산술/논리 연산을 수행하는 하드웨어 유닛

메모리 데이터 주소지정 (Addressing Data in Memory)

컴퓨터는 메모리를 선형 주소 공간(linear address space)으로 보고, 대부분 바이트 주소지정(byte-addressing)을 사용하며, 데이터는 워드 단위로 정렬된다.
상세 설명

컴퓨터가 메모리를 관리하는 방식의 기본 원칙:

  • 선형 주소 공간: 메모리는 0번지부터 시작하는 연속된 주소로 구성된다.
  • 바이트 주소지정(Byte-addressing): 대부분의 컴퓨터에서 최소 주소지정 단위는 1바이트(8비트)이다. 즉, 각 바이트마다 고유한 주소를 가진다.
  • 워드 정렬(Word alignment): 데이터는 워드(word) 단위로 정렬된다. 하나의 워드는 여러 바이트로 구성되며, 워드의 시작 주소는 워드 크기의 배수이다.

예를 들어, 4바이트 워드 시스템에서:

  • 주소 0, 4, 8, 12, ... 에서 워드가 시작된다.
  • 주소 0에는 바이트 0~3이, 주소 4에는 바이트 4~7이 속한다.
핵심 용어
영어한국어설명
Byte-addressing바이트 주소지정메모리의 최소 주소 단위가 1바이트인 방식
Word alignment워드 정렬데이터가 워드 크기의 배수 주소에서 시작하도록 배치
Linear address space선형 주소 공간메모리를 연속된 주소의 1차원 배열로 보는 개념

데이터 저장 방식: 엔디안 (How is Data Stored in Memory?)

32비트 데이터를 메모리에 저장할 때, 바이트 순서에 따라 리틀 엔디안(Little Endian)과 빅 엔디안(Big Endian) 두 가지 방식이 있다.
상세 설명

32비트(4바이트) 데이터는 Byte 0(MSB 쪽) ~ Byte 3(LSB 쪽)으로 구성된다. 이 4개의 바이트를 메모리에 어떤 순서로 배치하느냐에 따라 엔디안이 결정된다:

리틀 엔디안(Little Endian): LSB(Least Significant Byte)가 낮은 주소에 저장된다.

  • 주소 0: Byte 0, 주소 1: Byte 1, 주소 2: Byte 2, 주소 3: Byte 3
  • Intel x86 계열이 리틀 엔디안을 사용한다.

빅 엔디안(Big Endian): MSB(Most Significant Byte)가 낮은 주소에 저장된다.

  • 주소 0: Byte 3, 주소 1: Byte 2, 주소 2: Byte 1, 주소 3: Byte 0
  • 네트워크 프로토콜 등에서 빅 엔디안이 사용된다.

슬라이드 퀴즈 풀이:

메모리에 5개의 32비트 데이터가 주소 0, 4, 8, 12, 16에 저장되어 있다고 가정한다.

  • "녹색 데이터의 Byte 1 주소는?" — 녹색 데이터가 주소 16에서 시작한다면:
    • 리틀 엔디안: Byte 1은 주소 17
    • 빅 엔디안: Byte 1은 주소 18
  • "노란 데이터의 Byte 3 주소는?" — 노란 데이터가 주소 8에서 시작한다면:
    • 리틀 엔디안: Byte 3은 주소 11
    • 빅 엔디안: Byte 3은 주소 8
주의: 엔디안 방식은 바이트 단위의 순서이다. 비트 단위가 아니다. 또한 같은 데이터라도 엔디안이 다르면 메모리 덤프가 다르게 보이므로, 네트워크 통신이나 파일 교환 시 엔디안을 반드시 확인해야 한다.
핵심 용어
영어한국어설명
Little Endian리틀 엔디안LSB를 낮은 주소에 저장 (Intel x86 등)

레지스터 데이터 주소지정 (Addressing Data in Registers)

레지스터는 이름이나 인덱스로 주소지정되며, 초기에는 특수 목적이었으나 현재는 범용 레지스터(GPR)가 주류이다.
상세 설명

레지스터는 메모리와 달리 이름(name)이나 인덱스(index)로 접근한다. 예: $s1, $s2, $s3.

레지스터의 역사적 변화:

  • 초기: 특수 목적 레지스터(Special-purpose register)
    • 프로그램 카운터(PC)나 복귀 주소(return address) 저장 등 특정 용도로만 사용
    • 주소 레지스터(address register)에 대한 산술 연산은 지원되지 않았음
  • 현재: 범용 레지스터(General Purpose Register, GPR)
    • 주소 저장뿐 아니라 일반 산술 연산에도 사용 가능
    • 레지스터의 유연성이 크게 향상됨
핵심 용어
영어한국어설명
GPR범용 레지스터General Purpose Register, 다양한 용도로 사용 가능한 레지스터
Special-purpose register특수 목적 레지스터특정 기능에만 사용되도록 설계된 레지스터
Page 24

레지스터를 이용한 메모리 주소지정 (1)

메모리 주소를 지정하는 방식에는 직접 주소, 레지스터 간접 주소, 변위 주소 등이 있으며, 레지스터를 활용하여 메모리 데이터에 접근한다.
상세 설명

메모리에 데이터를 저장(store)하는 예를 들어 주소지정 방식을 설명한다. sw는 "store word"의 약자이다.

1. 직접 주소(Most basic / Absolute addressing)

sw $s1, 10000

$s1의 값을 메모리 주소 10000에 저장한다. 주소가 명령어에 직접 기술된다.

2. 레지스터 간접 주소(Register indirect)

sw $s1, ($s0)

$s0에 저장된 값을 주소로 사용하여, 해당 메모리 위치에 $s1의 값을 저장한다. 예: $s0 = 4이면 메모리 주소 4에 저장.

3. 변위 주소(Displaced address / Base + Offset)

sw $s1, offset($s0)

$s0 + offset을 주소로 사용한다. 배열이나 구조체의 특정 필드에 접근할 때 유용하다.

핵심 용어
영어한국어설명
Register Indirect레지스터 간접 주소레지스터에 저장된 값을 메모리 주소로 사용
Displaced (Base+Offset)변위 주소레지스터 값 + 상수 오프셋을 메모리 주소로 사용
Page 25

레지스터를 이용한 메모리 주소지정 (2)

인덱스 주소와 메모리 간접 주소 등 더 복잡한 주소지정 방식을 살펴본다.
상세 설명

4. 인덱스 주소(Indexed address)

sw $s1, ($s0+$s2)

$s0 + $s2를 주소로 사용한다. 배열 접근에 유용하다. $s0이 배열의 시작 주소, $s2가 인덱스 오프셋이라고 생각하면 된다.

5. 메모리 간접 주소(Memory indirect)

sw $s1, (($s0))

$s0에 저장된 값을 주소로 사용하여 메모리에서 다시 한 번 주소를 읽어온 뒤, 그 주소에 $s1의 값을 저장한다. C 언어의 포인터(pointer)와 유사하다.

예: $s0 = 4이고, Memory[4]에 12가 저장되어 있다면, 최종적으로 메모리 주소 12에 $s1의 값이 저장된다.

시험 포인트: 각 주소지정 방식의 차이를 명확히 구분하라. 특히 "레지스터 간접"은 1단계 간접 참조, "메모리 간접"은 2단계 간접 참조라는 점을 기억하자.
Page 26

메모리 vs 레지스터 (Memory vs Register)

메모리는 용량이 크지만 느리고, 레지스터는 빠르지만 비싸고 수가 적다. 이 트레이드오프가 컴퓨터 설계의 핵심이다.
상세 설명

메모리(Memory):

  • 다양한 매체로 구현 가능: DRAM(1T1C), SRAM(6T) 등
  • 보통 데이터를 읽고 쓰는 데 1 클럭 사이클 이상 소요
  • 레지스터보다 느리지만, 훨씬 많은 데이터를 저장할 수 있음
  • 상대적으로 저렴(cheap)

레지스터(Register):

  • 주로 플립플롭(Flip-flop)으로 구현
  • 한 클럭 사이클 안에 데이터를 소비하고 생산할 수 있음
  • 매우 빠르지만, 컴퓨터에서 수가 매우 적음 (희소 자원)
  • 비싸다: 1비트 레지스터(플립플롭)에 수십 개의 트랜지스터가 필요

비유: 레지스터는 작업대 위의 메모장(빠르게 접근, 공간 제한), 메모리는 서류 캐비닛(많이 저장, 접근에 시간 소요)이다.

핵심 용어
영어한국어설명
DRAM동적 RAM1개 트랜지스터 + 1개 커패시터(1T1C)로 구성, 대용량에 적합
SRAM정적 RAM6개 트랜지스터(6T)로 구성, 빠르지만 면적이 큼
Flip-flop플립플롭1비트를 저장하는 순차 회로, 레지스터의 기본 단위
핵심: 레지스터는 빠르지만 비싸고 부족하며(expensive & scarce), 메모리는 느리지만 크고 저렴하다(slow but large & cheap). 이 트레이드오프가 캐시 등 메모리 계층 구조의 근본 원인이다.

컴퓨터 속 숫자 (Numbers in Computer)

컴퓨터는 디지털 세계에서 프로그램을 실행하며, 모든 데이터를 이진수(binary)로 처리한다. 동일한 이진 데이터도 명령어에 따라 다르게 해석될 수 있다.
상세 설명

컴퓨터는 모든 것을 이진수로 표현한다. 중요한 것은 같은 비트열이라도 명령어에 따라 다른 형식으로 해석될 수 있다는 점이다.

정수(Integer) 형식의 세분화:

  • 바이트(byte): 8비트 (비트 0~7)
  • 하프워드(halfword): 16비트 (비트 0~15)
  • 워드(word): 32비트 (비트 0~31)

"워드(word)"의 정확한 정의는 무엇일까? 이것은 다음 페이지에서 자세히 설명한다.

Page 28

워드(Word)란 무엇인가?

워드(Word)는 특정 설계에서 데이터의 통상적인 길이를 나타내며, 기계에 따라 정의가 다르다. 32비트 머신은 4바이트, 64비트 머신은 8바이트가 1워드이다.
상세 설명

"워드(word)"라는 용어는 특정 설계에서 데이터의 일반적인 길이를 나타내는 데 사용된다.

  • 32비트 머신: 1 워드 = 4바이트 = 32비트
    • 바이트 주소 간격: 0, 4, 8, 12, ...
  • 64비트 머신: 1 워드 = 8바이트 = 64비트
    • 바이트 주소 간격: 0, 8, 16, 24, ...

명시적으로 크기가 지정되지 않은 경우, 명령어는 보통 1워드 단위로 동작한다.

주의: "워드"의 크기는 절대적인 값이 아니다. 반드시 해당 아키텍처의 정의를 확인해야 한다. 예를 들어, x86에서는 역사적 이유로 "word"가 16비트를 의미하기도 한다.
Page 29

부호 있는 데이터 표현 (Representing Signed Data)

컴퓨터에서 음수를 표현하기 위해 MSB를 부호 비트로 사용하며, 1의 보수(1's complement)와 2의 보수(2's complement) 방식이 있다.
상세 설명

실세계에서는 +/- 기호를 자연스럽게 사용하지만, 컴퓨터는 0과 1만 다루므로 부호를 표현할 규칙이 필요하다. 일부 ISA는 부호 있는(signed)/없는(unsigned) 수에 대해 서로 다른 명령어 동작을 정의한다.

기본 규칙: 최상위 비트(MSB)를 부호 비트(sign bit)로 사용. 0은 양수(+), 1은 음수(-).

3비트 부호 있는 정수 표현 비교:

10진수1의 보수2의 보수

1의 보수(1's complement): 모든 비트를 반전시켜 음수를 표현한다. 문제점: 0이 두 가지(000, 111)로 표현되어 숫자 하나를 낭비한다.

2의 보수(2's complement): 1의 보수에 1을 더하여 음수를 표현한다. 0이 하나뿐이고, -4까지 표현 가능하여 1의 보수보다 1개 더 많은 음수를 표현할 수 있다.

2의 보수가 현대 컴퓨터에서 표준으로 사용되는 이유: 0의 표현이 유일하고, 덧셈 회로 하나로 덧셈과 뺄셈을 모두 수행할 수 있기 때문이다.
시험 포인트: 1의 보수에서 0이 두 개인 것("Wasting a number!")이 왜 문제인지, 2의 보수가 왜 우월한지 설명할 수 있어야 한다.

부동소수점 데이터 (Floating Point Data)

부동소수점(Floating Point)은 실수를 표현하는 방식이며, IEEE 754 표준이 업계 합의를 통해 사용된다.
상세 설명

부동소수점(Floating point)은 실수(real number)를 표현하기 위한 방법이다. 정수만으로는 소수점이 있는 수(예: 3.14)를 표현할 수 없으므로, 별도의 형식이 필요하다.

IEEE 754: 부동소수점 수의 표현에 대한 국제 표준. 부호 비트, 지수(exponent), 가수(mantissa/significand)로 실수를 표현한다. 업계 전체가 이 형식에 합의했다.

부동소수점 연산을 수행하는 두 가지 방법:

  1. 정수 명령어 여러 개를 사용하여 부동소수점 연산을 소프트웨어적으로 구현 (예: 정수 명령어들로 float-add를 구현)
  2. 부동소수점 전용 명령어를 ISA에 설계하여 하드웨어에서 직접 처리

전용 명령어 방식이 훨씬 빠르지만, 하드웨어 복잡도가 증가한다는 트레이드오프가 있다.

CISC ISA (Complex Instruction Set Computer)

CISC는 가변 길이 명령어를 사용하여 풍부한 기능을 간결한 코드로 표현하지만, 디코딩이 복잡하고 지연 시간(latency)이 길다.
상세 설명

CISC(Complex Instruction Set Computer)의 특징:

  • 가변 명령어 길이(Variable instruction lengths): 명령어마다 길이가 다르다. 예: 어떤 명령어는 8비트, 다른 명령어는 32비트.
  • 포괄적인 기능 표현: 하나의 명령어로 복잡한 작업을 수행할 수 있어, 코드 크기(code size)가 작아진다.

CISC의 대표적 예: VAX-11 (DEC, 1977)

  • 연결 리스트(linked list) 전용 명령어 지원
  • 4차 다항식(4th-degree polynomial)을 풀기 위한 명령어 지원

단점:

  • 복잡한 디코딩 로직: 가변 길이이므로 명령어 경계를 파악하기 어려움
  • 높은 디코딩 복잡도로 인한 긴 지연 시간: 깊은 파이프라이닝(deep pipelining)이 필요
핵심 용어
영어한국어설명
CISC복합 명령어 집합 컴퓨터복잡하고 다양한 명령어를 지원하는 ISA 방식
Variable instruction length가변 명령어 길이명령어마다 비트 수가 다른 형식
Pipelining파이프라이닝명령어 실행 단계를 겹쳐서 처리량을 높이는 기법

RISC ISA (Reduced Instruction Set Computer)

RISC는 고정 명령어 크기와 레지스터-레지스터 연산을 사용하여 디코딩을 단순화하고, 빠른 클럭과 작은 면적을 실현한다. 학계에서 탄생한 혁신적인 ISA 철학이다.
상세 설명

RISC(Reduced Instruction Set Computer)의 특징:

  • 고정 명령어 크기(Fixed instruction size): 모든 명령어가 동일한 비트 수를 가짐
  • 레지스터-레지스터 연산(Register-to-register operation): 연산은 레지스터 사이에서만 수행. 메모리 접근은 별도의 load/store 명령어가 담당.
  • 단순한 디코딩 복잡도 → 빠른 클럭 속도, 작은 칩 면적

RISC의 역사:

  • 최초의 RISC 제품: IBM 801
  • 학계(academia)에서 발명된 것이 특징 — 학계의 중요성을 보여주는 사례!
    • Berkeley RISC: David Patterson 교수가 주도
    • Stanford MIPS: John Hennessy 교수가 주도

단점:

  • CISC에 비해 코드 크기가 커질 수 있다. 간단한 작업도 여러 명령어가 필요할 수 있기 때문.
핵심 용어
영어한국어설명
RISC축소 명령어 집합 컴퓨터단순하고 균일한 명령어를 사용하는 ISA 방식
Fixed instruction size고정 명령어 크기모든 명령어의 비트 길이가 동일한 형식
Load/Store architecture적재/저장 구조메모리 접근은 load/store만, 연산은 레지스터에서만 수행
시험 포인트: David Patterson(Berkeley RISC)과 John Hennessy(Stanford MIPS)는 이 교재의 저자이기도 하다. 두 사람은 RISC 개념의 창시자로, 2017년 튜링상을 수상했다.

RISC가 주목받는 이유 (RISC: Why Is It Gaining Traction?)

현대 CISC 프로세서도 내부적으로 RISC 스타일의 마이크로코드를 사용하며, 메모리 속도 향상과 컴파일러 최적화 용이성이 RISC의 강점을 더욱 부각시킨다.
상세 설명

RISC가 점점 더 주목받는 세 가지 이유:

1. CISC 명령어도 내부적으로 RISC처럼 처리된다

  • CISC 명령어는 MIPS와 유사한 마이크로코드(micro-code)로 디코딩된다.
  • 마이크로코드를 ROM에 저장하여, CISC 프로세서(Intel, AMD 등)도 내부적으로는 단순한 디코딩 로직을 사용한다.
  • 버그 수정 시 칩을 다시 제작(re-tape)하지 않고도 ROM의 마이크로코드만 패치할 수 있어 유리하다.

2. 메모리가 점점 빨라지고 있다

  • 메인 메모리와 캐시의 속도가 향상되면서, RISC의 코드 크기 증가(더 많은 명령어)가 문제가 되지 않게 되었다.

3. RISC는 컴파일러 최적화에 유리하다

  • RISC는 단순하기 때문에 하드웨어의 세부 사항이 더 많이 노출(expose)된다. 이는 컴파일러가 최적화하기 쉽다는 의미이다.
  • 반면, CISC ISA의 간결함(compactness)은 하드웨어의 많은 세부 사항을 숨기기(hide) 때문에, 컴파일러 최적화가 어렵다.
  • 예: 나눗셈의 중간 결과를 프로그래머에게 공개해야 할까? RISC 철학에서는 "예"라고 답하여, 컴파일러가 더 정교한 최적화를 수행할 수 있다.
현대 CPU 설계의 핵심 교훈: 외부적으로는 CISC ISA(예: x86)를 유지하면서, 내부적으로는 RISC 스타일의 마이크로코드로 변환하여 실행하는 하이브리드 방식이 주류이다. Intel/AMD의 x86 프로세서가 대표적인 예이다.

인라인 어셈블리 (Inline Assembly)

인라인 어셈블리는 고수준 언어(C/C++) 안에서 직접 어셈블리 코드를 작성하는 기법이다. ISA의 어셈블리와는 다르며, 특정 컴파일러가 인식하는 형태로 작성된다.
상세 설명

인라인 어셈블리(Inline Assembly)란?

  • 고수준 언어 코드 안에서 어셈블리를 직접 작성하는 방법
  • C/C++ 표준의 일부가 아님. 특정 컴파일러(예: GCC)가 인식하는 확장 기능.
  • 컴파일러는 이 "고수준 어셈블리"를 독립적인 어셈블리 파일(*.s)로 변환한다.
  • 컴파일러가 고수준 어셈블리를 저수준 어셈블리로의 매핑 정보를 가지고 있다.

사용 예시 — C 코드에서 인라인 어셈블리:

static unsigned long GetRDTSC() {
    unsigned Low, High;
    __asm__("rdtsc" : "=a"(Low), "=d"(High));
    return ((unsigned long)Low) |
           (((unsigned long)High) << 32);
}

위 코드는 x86의 rdtsc (Read Time-Stamp Counter) 명령어를 C 함수 안에서 직접 사용하는 예이다. __asm__ 키워드가 GCC에게 인라인 어셈블리임을 알려준다.

컴파일된 독립 어셈블리 파일의 동일 기능:

.global GetRDTSC
GetRDTSC:
    rdtsc
    shl $32, %rdx
    or %rdx, %rax
    ret

사용 시 주의점:

  • 일반적으로 사용이 권장되지 않음 (이식성 문제, 디버깅 어려움)
  • 하지만 하드웨어 제어, 성능 측정 등 반드시 필요한 경우가 있음
